# Git compatibility gap closure plan (2026-07-08)

## 文档职责

本文把 `/Users/eli/Downloads/libra_git.md` 中的 Git 对比探针结果（Libra `0.18.21` vs Git `2.55.0`，~940 条记录）收敛成 Libra 的兼容改进计划，并纳入两项与探针独立、但同属“命令表面可靠性/体验”的 Issue：**#434（全局 config 库 schema 比二进制新时 `pull` 静默降级）**、**#437（安装时可选 `lba` 别名）**。

目标不是 100% 复刻 Git 的长尾 surface，而是优先修复会导致**静默错误、历史形态漂移、CI/脚本误判、用户漏看冲突、配置被无声忽略**的高风险差距。

本文只规划任务，不宣称实现完成。落地时每个任务都必须同步：`COMPATIBILITY.md`、`docs/development/commands/_compatibility.md`、对应 `docs/development/commands/<cmd>.md`（含 EN↔zh 双语 parity）、用户命令文档、`tests/INDEX.md`、`Cargo.toml` 的 `[[test]]` target、`docs/error-codes.md`（如新增 `LBR-*`）以及 compat/integration 测试。

### Issue 基线（2026-07-08 核对）

- [#434](https://github.com/libra-tools/libra/issues/434) 当前为 **OPEN**，label `experience`，milestone `1.0.0`。问题是旧二进制读取由新二进制迁移过的全局 config DB 时，`libra pull` 在 schema-newer 错误后继续 fallback 到 local storage；期望是 fail-fast 或显式、可理解的降级诊断。
- [#437](https://github.com/libra-tools/libra/issues/437) 当前为 **OPEN**，label `experience`，milestone `1.0.0`。需求是在安装目录内创建短别名 `lba -> libra`，可自动创建或由安装器 flag 控制。

### 规格基线（Spec baseline）

- **Git 参照版本固定为 2.55.0**（探针环境）。“Git 兼容”一律指与该版本的 documented behavior 对齐，而不是一个漂移的移动目标。每个 machine-output 任务须在测试里以 golden fixture 或（CI 存在真实 `git` 时）真机 diff 锚定，不用口语化的“对齐 Git”当验收标准。
- 权威规格来源：Git man pages 与 `gitformat-*`（`gitformat-index`、`git-status(1)` porcelain v1/v2、`git-diff(1)` `--check`/exit-code、`git-grep(1)` exit-code 等）。任务描述中的“精确验收”即引用这些规格的具体输出/退出码。
- 探针文件是**证据来源**：每个任务在“证据”里回指 `libra_git.md` 的条目号（`§N`）；标注为 *derived* 的任务是从探针行为推导的加固项，非单条探针，需在实现前先复现实证。

## 评审结论与本次修订（Review summary）

对上一版计划做了合理性/可行性/完整性/安全性/功能正确性与接口兼容性/数据流与控制流/性能/可靠性容错/兼容互操作/可扩展可维护/合规标准 共 11 个维度的评审，发现并在本次修订中补齐的系统性缺口如下：

1. **验收标准太软**（功能正确性）：原文多为“对齐 Git”，无法直接判定完成。→ 每个 machine-output 任务补“精确验收”（确切 XY 码 / 记录格式 / 退出码 / golden fixture）。
2. **缺工程横切约束**（完整性/安全/性能/互操作）：hash-kind(sha1/sha256) 中立、命令执行安全（textconv/filter/hooks 走 sandbox）、fail-closed 策略、共享引擎增量接入、热路径性能预算、真机 Git 互操作、EN↔zh 文档 parity、compat-surface 守卫全部缺失。→ 新增“全局工程约束 GC-01..GC-11”。
3. **大任务未拆分**（可行性）：`pathspec engine`、`import/export`、`config defaults` 是跨多命令的大改，按单任务估算不现实。→ 拆成可独立落地、可并行、可回归的子步骤，并要求逐消费者 parity，不做 flag day。
4. **未做现状复核**（可行性）：部分缺口在 `0.18.22` 已部分收敛（如 `cat-file` 已处理 `BrokenPipe`，但无全局策略）。→ 每个任务加“现状核实”前置动作，避免重复劳动/误报完成。
5. **两个 Issue 未纳入**（完整性）：#434、#437 未在计划内。→ #434 归入 P0（静默降级=安全类），#437 归入新的 IX 安装/人体工学轨。
6. **无追溯与风险登记**（可维护/可靠性）：无 task↔probe↔compat↔test 映射，无风险登记。→ 新增追溯表与风险登记。
7. **回归面被低估**（可靠性/接口兼容）：多数改动会改变现有 libra 测试和 compat-surface 守卫断言的输出。→ 验收门禁显式列出必须同步更新的守卫，并要求每个任务评估“影响面/回归”。

## 全局工程约束（Cross-cutting engineering constraints）

以下约束对**所有**任务生效，任务条目不再逐条重复；违反任一项即视为未完成。

- **GC-01 现状核实前置**：动手前先在当前 `main` 复现探针结论（部分缺口可能已收敛）。若已实现，则任务降级为“补测试 + 补文档 + 兼容矩阵登记”，不重写。
- **GC-02 hash-kind 中立**：凡读写对象/index/tree/ref 的任务，必须走 `cli.rs` 的 hash-kind 预检（`core.objectformat` = `sha1`/`sha256`），不得硬编码 20/32 字节 OID 长度；测试至少覆盖 sha1，且对新增 plumbing 增加一条 sha256 冒烟。
- **GC-03 命令执行安全**：任何会执行外部命令的表面（`diff --textconv`/`cat-file --textconv`/attributes filter、`.libra/hooks` 生命周期、`rebase --exec`）必须经过 `internal::ai::sandbox` 的 command-safety policy / seatbelt-SBPL 审查路径，默认不在无审查下执行用户可控命令；相关任务须在描述里显式声明其信任边界。
- **GC-04 fail-closed 默认**：当命令语义依赖的前置（对象闭包、全局存储 config、schema 兼容）无法满足时，默认 fail-closed（清晰报错 + 不产出误导性成功摘要 + 不留下 broken 状态），仅在明确、可观测的降级路径下 fail-open。禁止“打印成功但语义未执行”。
- **GC-05 共享引擎增量接入**：`pathspec`/`ignore+attributes`/`pretty-format` 抽成真正共享模块（建议 `src/utils/pathspec/`、`src/utils/attr/`、`src/internal/log/pretty/`），逐消费者切换并各配 parity 测试，不做一次性 flag-day 替换；切换某命令前，其旧行为的现有测试必须保留或显式迁移。
- **GC-06 输出/退出码/错误码纪律**：机器输出必须同时满足 `--json`/`--machine`（`utils::output::OutputConfig`）与人读两条路径；用户可见错误一律用稳定 `LBR-*` 码（`utils::error`），并同步 `docs/error-codes.md`（`compat_error_codes_doc_sync` 守卫会拦截漏登记）。退出码对齐 Git（见各任务“精确验收”）。
- **GC-07 原子性与回滚**：改 HEAD/refs/index/仓库布局的任务必须“要么完整生效、要么完全不动”；失败路径不得移动 HEAD、不得留下半成品 `.libra` 或半更新的 ref。
- **GC-08 性能预算**：不得回归 `commit`/`status`/`diff`/`add` 热路径。新增的逐路径校验（如 `write-tree` object 存在性、attributes 匹配）须 O(变更集) 而非 O(全树)，attributes/ignore 匹配要有编译期缓存；对 >10k entry 的仓库补一条性能不回归基准（criterion 或 wall-clock 断言）。
- **GC-09 真机 Git 互操作**：machine-output 与 import/export 任务在 CI（`[self-hosted]` 有真实 `git`）下增加真机对照/round-trip；无 `git` 环境用 `env_var_is_set`/工具探测 gate 并打印 skipped，绝不硬失败。
- **GC-10 文档 EN↔zh parity**：命令文档与 `_compatibility.md` 的中英内容必须同步；历史经验表明 zh-CN 文档常严重滞后，实现前先审计 parity（`compat_command_docs_examples_section` 等守卫会拦截缺 Examples 段）。
- **GC-11 接口冲突预检**：新增 flag 前先核对与既有 Libra-only flag / 全局 flag（如 `--progress`）的冲突（参照 `_compatibility.md` 已记录的 `restore --progress` DEAD 教训），避免 clap 层撞名。

### 命令开发基础要求（内联版）

以下规则来自 `docs/development/commands/_general.md` 的基础要求，本文直接内联为本计划的执行前置；后续任务不得只引用外部文档而跳过这些约束。

- **命令事实源**：公开命令以 `src/cli.rs::Commands` 为入口事实源，以 `src/command/mod.rs` 为模块导出事实源；未出现在公开 enum 中的文档页不得被描述为可用 CLI。
- **兼容承诺事实源**：Git 兼容命令以 `COMPATIBILITY.md` 为用户承诺，以 `docs/development/commands/<cmd>.md` 记录实现细节、历史、当前状态和未完成项；集成验证以 `docs/development/integration/integration-test-plan.md` 与 `docs/development/integration/integration-scenarios/integration-scenarios.yaml` 为事实源。
- **Libra 扩展命令说明**：`code`、`agent`、`cloud`、`publish`、`usage`、`sandbox` 等 Libra 扩展命令不追求 Git 同形，但必须说明与 Git/通用 CLI 的差异、替代工作流、用户影响和重启条件。
- **全局 CLI 契约**：所有命令改动必须核对 `--json`、`--machine`、`--no-pager`、`--color`、`--progress`、`--quiet`、`--exit-code-on-warning` 的行为；这些 flag 是 Agent 驱动 CLI 的基础契约，不能只改人读输出。
- **全局耐久性契约**：涉及本地对象、index、ref、sequencer 或恢复关键状态的写入，必须说明 `--sync-data` / `LIBRA_SYNC_DATA=1` 的影响；recovery-critical 状态即使未启用该开关，也必须按现有 `utils::atomic_write` / fsync 约束保持可恢复。
- **全局资源契约**：涉及远端连接、批量对象检查、网络请求或 fan-out 的改动，必须说明 `--max-connections <N>` / `LIBRA_MAX_CONNECTIONS` 的优先级和消费点；非法 env 值按 usage 错误退出，纯本地操作应为 no-op。
- **全局取数策略契约**：涉及分层对象存储或远端读取的命令，必须遵守 `--offline` 与 `LIBRA_READ_POLICY=auto|offline|local|remote` 的优先级和 fail-closed 语义；不可新增与 `config --local`、`clone --local`、`agent push --remote` 等现有表面冲突的 `--local`/`--remote` 逃逸阀。
- **命令文档固定结构**：每个命令开发文档必须维护六段：实现目标、Git 兼容性、设计方案、实现历史、当前状态、未完成项；`README.md` 只作为索引，不承载单命令设计。
- **文档更新流程**：新增或修改命令文档前，先核对 `src/cli.rs`、`src/command/<cmd>.rs` 和必要子模块，再把入口、参数类型、输出/错误类型、执行流程、测试证据写回对应文档；Git 兼容命令还必须同步 `COMPATIBILITY.md`、集成计划、场景 YAML 和 runner 证据。
- **副作用边界**：涉及 `.libra/index`、对象库、SQLite refs/HEAD/reflog、D1/R2、Agent session/checkpoint、worktree registry 的命令，文档必须点名读写对象、事务边界、失败回滚和清理行为。
- **输出与错误契约**：每个命令页都必须说明 `CliError` / `CliResult`、`OutputConfig`、JSON/machine 输出、stable error code 的维护方式；未公开命令必须显式标记为未发布。
- **维护要求**：新增命令必须同时补 `src/cli.rs`、`src/command/mod.rs`、实现模块、用户文档、开发文档、`COMPATIBILITY.md` 和测试；新增参数必须说明 Git 对齐情况、JSON/机器输出影响、错误码和回归测试；移除或延后能力必须在命令文档和 `_compatibility.md` 中说明用户影响与重启条件。
- **跨命令守卫**：命令文档或兼容承诺改动后，至少运行 `cargo test --test compat_matrix_alignment` 与相关 compat 测试，防止文档、集成测试方案和代码状态再次分叉。

### 逐项改进执行与发布门禁

- **ER-01 先核对后执行**：每一个改进开始前，先把计划、对应开发文档、用户命令文档和当前代码实现逐项核对；如果代码已实现、文档滞后或计划已过期，先把该任务调整为“补测试 / 补文档 / 降级非目标 / 继续实现”的最新状态，再开始编码。
- **ER-02 Libra-native 工作流**：本地版本管理使用 Libra，不把仓库当 Git 仓库处理。执行状态、暂存、提交、推送时使用 `libra status`、`libra add`、`libra commit`、`libra push`；不要用 Git-only 分支/worktree 假设推导状态。
- **ER-03 单分支执行**：所有改进都在 `main` 分支直接执行，不创建 worktree，不创建 feature branch；开工前确认当前分支是 `main`，并在脏工作区中只暂存当前改进相关路径。
- **ER-04 每项验收三门**：每一个改进完成前必须全部通过 `cargo +nightly fmt --all --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`source .env.test && cargo test <该改进指定测试用例>`。任务条目必须写出指定测试 target / test fn；不能用“后续补测”替代。
- **ER-05 Codex review 闭环**：每一个改进完成本地测试后，交由 Codex 进行 review；review 发现的问题必须修复并重新验收，直到 Codex review 通过，才允许进入版本发布步骤。
- **ER-06 用户命令文档同步**：每一个涉及命令表面的改进，都必须更新 `docs/commands` 下对应英文文档和中文文档；若对应命令没有文档，则创建 EN 与 zh-CN 两份用户文档，并同步开发文档、compat 矩阵和测试索引。
- **ER-07 AI 测试模型**：凡设计为调用 AI 服务的测试，统一使用本地 Ollama 模型 `deepseek-v4-pro:cloud`；不得把外部商业 API key 作为默认验收前置。
- **ER-08 每项发布 patch 版本**：每一个改进验收和 Codex review 通过后，将当前版本 patch 加 1（例：`version = "0.17.500"` → `version = "0.17.501"`），同步 `Cargo.toml`、`web/package.json`、`worker/package.json` 的版本号；不要手工修改 `Cargo.lock`，而是执行 `cargo build --release` 让工具链更新。
- **ER-09 本机安装与提交推送**：版本构建完成后，把 release 产物 `target/release/libra` 拷贝到 `$HOME/.libra/bin/libra`；随后按 Libra 工作流执行 `libra add <相关路径>`、`libra commit -s -m "<scope>: <summary>"`、`libra push origin main` 推送到 GitHub。**不使用 `commit -a`**（并发节点/带外改动会被误暂存，见「执行检查必备需求」第 6 条，本条以其为准）。
- **ER-10 push 失败策略**：如果 `libra push origin main` 失败，不重试该次 push；记录失败原因，到下一次修复完成并通过验收、review、版本构建后，再随下一次发布尝试 push。
- **ER-11 内部服务错误策略**：除 push 失败外，遇到 AI provider、MCP/connector、R2/D1、GitHub API、release/download 等内部服务的暂时性错误时，不暂停目标、不把任务标记为完成或阻塞；记录错误，等待 10 分钟后重试同一步，直到任务重新开始。若错误被证实是确定性代码/配置缺陷，则转为当前改进的修复项。

## 执行检查必备需求（强制）

以下要求适用于本文每一个改进项（A0 / CG / P0 / P1 / P2 / IX 全部）。任一要求未满足，该改进不得标记完成，不得进入下一项发布流程。本节是本计划**唯一**强制的发布 / review / push / 服务错误流程，与上文 **ER-01..ER-11** 及「验收门禁」语义对齐并**统一到本节**；两处存在差异时以本节为准（尤其 commit 方式，见第 6 条）。本节与 `plan-20260713.md` 的「执行检查必备需求（强制）」保持一致，便于两台机器并发协作、共用同一套发布/重试语义。

1. **每一个改进验收必须三项全部通过**：`cargo +nightly fmt --all --check` 无格式差异；`cargo clippy --all-targets --all-features -- -D warnings` 无警告；`source .env.test && cargo test <指定测试用例>` 全部通过。任务卡里的指定测试用例必须在开工前写清楚，不能用泛泛的 `cargo test` 替代。

4. **本地版本管理使用 Libra**：本仓库本地使用 Libra 作为版本管理，不要当做 Git 仓库处理；状态、暂存、提交、推送都使用 `libra` 命令和 Libra-native 工作流。
5. **所有改进都在 `main` 分支进行**：不要创建 worktree，不要创建 feature branch；开工前确认当前分支是 `main`，并在脏工作区中只处理当前改进相关路径。
6. **每一个改进完成后发布 patch 版本并推送**：每一个改进验收和 Codex review 通过后，对当前版本 patch 加 1（以开工时 `Cargo.toml` 实际 `version` 为准，如 `0.18.NNN` → `0.18.(NNN+1)`，不要硬编码某个历史号）；`worker` 和 `web` 子目录下的 `package.json` 版本必须同步保持一致。不手工修改 `Cargo.lock`；版本修改后执行 `cargo build --release`，让工具链更新锁文件。构建出的 release 版本 `target/release/libra` 必须拷贝到 `$HOME/.libra/bin/libra`。随后执行 `libra add <相关路径>`、`libra commit -s -m "<scope>: <summary>"`、`libra push origin main` 推送到 GitHub。**不要使用 `commit -a`**：本仓库工作区常有并发节点改动（`plan-20260713.md` 正在另一台机器并发执行）与带外删除，`-a` 会把非本任务路径一起提交（与第 5 条冲突）；一律用显式 `libra add <相关路径>` 后 `libra commit -s -m`（不加 `-a`）。**本条取代 ER-09 与「验收门禁」中 `commit -a -s` 的旧写法。**

### 版本抢号与 push 冲突解决（与 `plan-20260713.md` 并发共享 `main`/版本号）

`plan-20260713.md` **正在另一台机器并发执行**，与本计划**争夺同一个 `Cargo.toml` 单调版本号**并推送同一 `origin/main`。两个计划据此共用**抢号模型**：版本号在 **push 成功时**才算领到，不在本地 commit 时锁定；遇到 origin 已领先就整合、重新领号、再推。本节与 `plan-20260713.md` 的同名小节内容一致，两台机器须遵循相同协议。

- **push 冲突流程**：当 `libra push origin main` 因 origin 版本更高/领先而失败（non-fast-forward，如 `LBR-CONFLICT-002`）时：
  1. `libra pull`（或 `fetch` + `merge`）整合 origin 的并发提交（本仓库 pull 有历史 hang 记录，须加超时/后台监控，超时则报告不静默卡死）；
  2. **按语义解决冲突**（见下表），不做纯文本合并；
  3. 合并后**重新领号并重跑验收**：版本改为 `max(origin, 本地) + 1`、`worker`/`web` 的 `package.json` 三处同步一致、`cargo build --release` 更新 `Cargo.lock`，再跑 fmt/clippy/指定测试确认合并未破坏构建；
  4. 再 `libra push origin main`；仍失败则重复本循环（用 tight `libra pull origin main && libra push origin main` 收窄竞态窗口）。**禁止 `--force`**——绝不丢弃 origin 的并发提交。
- **语义冲突解决策略**（共享登记/文档文件）：

  | 文件 | 语义解决 |
  |---|---|
  | `Cargo.toml` `version` + `worker/package.json` + `web/package.json` | 不做文本合并；取 `max(两侧) + 1` 为新号，三处同步一致，`cargo build --release` 更新 `Cargo.lock` |
  | `docs/error-codes.md` / `LBR-*` 码 | 取并集；若两侧用了**同一码号**，把本地的重排到下一个空闲码并更新全部引用（`src` + `docs` + `tests`） |
  | `sql/migrations/*` + `builtin_migrations()` + `tests/db_migration_test.rs` 钉死列表 | 取并集并保序；文件名日期/迁移 ID 冲突则给本地迁移重命名，再同步注册与钉死列表 |
  | `tests/INDEX.md` | 并集两侧 Wave 条目 |
  | `COMPATIBILITY.md` | 并集行；同一命令行冲突则合并 Notes |
  | `src/cli.rs` 命令组 / `command/*/mod.rs` | 并集命令变体 |
  | 其它源码 | 通常不同域（本计划 Git 兼容/A0 vs 20260713 agent 捕获），按各自意图合并；真正同区冲突按行为正确性解决，不机械取一侧 |
- **与第 6、9 条的关系**：本节**细化**而非放宽——第 9 条"push 失败不 blind-retry"仍适用于真正的 push 失败（认证/权限/网络/服务端：记录、不重试、随下次发布携带）；本节只处理 **origin 版本领先的 non-fast-forward**，"pull + 语义合并 + 重新领号 + 推送新状态"正是第 9 条要求的"修复→重新验收→版本构建后再尝试"，非重试同一 push。第 6 条"读开工实际版本 +patch"照旧，只是抢号下"实际版本"在每次冲突后**以 pull 后的 origin 值重新读取**再领下一个号。

### P0-01 修复 unmerged/conflict 状态与 diff 展示

- **问题**：merge/rebase/cherry-pick 冲突路径在 `status --porcelain` 中被报成 `??`，`diff` 把冲突文件当作从 `/dev/null` 新增，脚本和用户漏看冲突。
- **证据**：`§16`（冲突报成 untracked）、`§40`（porcelain v2 XY 格式不同）、`§41`（rebase 冲突 exit/status）、`§83`（`ls-files -m` 不含已删 tracked）、`§200`（`ls-files -t` 标记不同）、`§5`（`rm --cached` 展示）、`§68`（`mv` rename 不入 porcelain）。
- **现状核实**：`src/command/status.rs` 当前无 `UU`/unmerged 输出（已确认缺失）。
- **范围**：`status`、`diff`、`ls-files -u/-t`、merge/rebase/cherry-pick sequencer 状态。
- **精确验收**：
  - `status --porcelain=v1`：对七类 unmerged 组合输出正确 XY 码 —— `DD`(both deleted)、`AU`(added by us)、`UD`(deleted by them)、`UA`(added by them)、`DU`(deleted by us)、`AA`(both added)、`UU`(both modified)。内容冲突 → `UU`。
  - `status --porcelain=v2`：输出 `u <XY> <sub> <m1> <m2> <m3> <mW> <h1> <h2> <h3> <path>` 形状的 unmerged record（stage1/2/3 mode 与 OID 齐全）。
  - `diff`：对 unmerged path 走 stage 视图（默认 combined/`--cc` 语义），至少**不得**把冲突误报为 `/dev/null` untracked add。
  - `ls-files -t` 不隐藏 unmerged entries；`ls-files -u` 输出 stage 行。
- **错误码**：与冲突态交互的阻断路径复用 `LBR-CONFLICT-001/002`。
- **影响面/回归**：改动 status/diff 主输出；同步更新现有 status/diff 测试与快照。
- **测试**：`tests/compat/conflict_status_diff_test.rs`，覆盖 merge、rebase、cherry-pick 三条路径 × v1/v2。

### P0-02 补齐 `diff --check` 的安全检查（derived 加固）

- **证据**：*derived*（探针主证是 `apply --check` 差异 `§15`；`diff --check` 的三类检查按 Git `git-diff(1)` 规格加固）。实现前先在 `main` 复现当前 `diff --check` 覆盖面。

- **精确验收**：至少覆盖 Git 三类检查并对每类输出 `<file>:<line>: <message>`：trailing whitespace、leftover conflict marker、new blank line at EOF；发现问题时**退出码 = 2**（与 Git 一致），干净时 0。
- **测试**：`tests/compat/diff_check_safety_test.rs`。

### P0-03 禁止产生 broken shallow clone

- **问题**：local `clone --depth 1` 可创建无法遍历、`fsck` 报 missing object 的仓库。
- **证据**：`§22`（`clone --depth 1` 本地源产生 broken shallow，其正文含浅克隆 `fsck --no-dangling` 报 missing object）、`§23`（`rev-parse --is-shallow-repository` 不支持）。
- **范围**：`clone`、fetch object closure、shallow metadata。
- **精确验收**：
  - 浅克隆对象闭包不完整时，命令必须失败（`LBR-REPO-002` 或新增 `LBR-CLONE-*`）并**回滚目标目录**或标记不可用（GC-07）。
  - 暂不支持的 shallow source 必须 fail-closed，不得留下 broken repo。
  - 成功的浅克隆 `fsck` 不得报 missing object；`rev-parse --is-shallow-repository` 至少返回正确布尔（可作为子项，非阻塞）。
- **执行决策（2026-07-09）**：现状复核确认 broken path 是本地 Libra 源 `fetch --depth` 会截断 commit walk 但不发送 `shallow <oid>`，导致 `clone --depth` 成功后缺父提交且无 `.libra/shallow`。本轮不假装完整 shallow 协议已支持，改为在 fetch 层对本地 Libra remote + `--depth` fail-closed（`LBR-REPO-002`），clone 继承该错误并在创建目标前/失败清理后不留下 initialized target；Git shallow 协商路径不扩大改动。同步实现 `rev-parse --is-shallow-repository` 只读 shallow 元数据返回布尔。
- **测试**：`tests/compat/clone_shallow_integrity_test.rs`；cargo target `compat_clone_shallow_integrity`。

### P0-04 修复 checkout 分支创建后 detached HEAD

- **问题**：`checkout -b <branch> <start-point>` / `checkout -B <branch> <start-point>` 可成功但留下 detached HEAD。
- **证据**：`§171`（`checkout -B` 不切到分支）。`§57`（`checkout -`/`switch -` 上一分支）经 2026-07-09 复核属于 previous-branch shortcut，不属于本项的 branch-create start-point / detached HEAD 失败路径；本计划新增「P1-12 previous-branch shortcut」单独治理，避免把未覆盖语义误报为 P0-04 已完成。
- **范围**：`checkout` 与 `switch` 分支创建共享逻辑。
- **精确验收**：`-b`/`-B` 创建/重置分支后 `HEAD` 为 symbolic ref 指向该分支（`rev-parse --symbolic-full-name HEAD` = `refs/heads/<branch>`）；失败时不得移动 HEAD（GC-07）。
- **执行决策（2026-07-09）**：源码核对确认 `switch -c/-C <name> [<start-point>]` 已走分支创建后切换路径；缺口集中在 `checkout -b/-B <branch> <start-point>`，原实现把 `<start-point>` 作为普通 checkout target 解析并成功 detached。修复策略为 checkout 分发优先识别 `-b/-B` 的可选 start-point，先解析并加载 commit、预检工作树，再创建/重置分支并切换到 symbolic `HEAD`；无效 start-point 保持原 HEAD 与既有分支引用不变。
- **测试**：`tests/compat/checkout_branch_startpoint_test.rs`；cargo target `compat_checkout_branch_startpoint`。

### P0-05 修复 orphan 分支语义

- **问题**：`switch --orphan` / `checkout --orphan` 可保留旧 parent，违背 root branch 语义。
- **证据**：`§58`（`checkout --orphan` 不支持）。
- **范围**：`switch --orphan`、`checkout --orphan`、index/worktree 初始化、后续 commit parent。
- **精确验收**：orphan 后首个 commit 无 parent（`rev-list --count HEAD` = 1、`log --pretty=%P -1` 为空）；index 保留但 HEAD 指向未生成的新分支；若无法保证 root 语义，命令 fail-closed 拒绝。
- **执行决策（2026-07-09）**：源码核对确认当前 `switch --orphan` 会立即生成 `.librakeep` 占位 root commit，导致命令本身已经产生提交；`checkout --orphan` 在当前 `CheckoutArgs` 中缺失。当前修复改为 unborn branch 语义：`switch --orphan` / `checkout --orphan` 仅把 `HEAD` 指向尚未生成 ref 的 `refs/heads/<branch>`，不创建占位对象/分支 ref，不清空工作树；index/worktree 保留，后续首个 `commit` 通过既有 unborn HEAD 路径生成无 parent root commit。同名分支、当前分支名、其它 worktree 已 checkout 的 unborn 名称、checkout/switch 额外 start-point 均 fail-closed，避免删除既有分支或静默忽略用户输入。
- **测试**：`tests/compat/switch_orphan_root_test.rs`；cargo target `compat_switch_orphan_root`。

### P0-06 处理 stdout broken pipe，不得 panic（收敛为全局策略）

- **问题**：`cat-file -p HEAD | head -n 1` 等正常 Unix pipeline 可触发 panic。
- **证据**：`§82`（`cat-file` 在 broken pipe 上 panic）。
- **现状核实**：`cat-file`/`pager`/`rev_list_output` 已各自处理 `ErrorKind::BrokenPipe`，但 `main.rs` **无全局 SIGPIPE/EPIPE 策略**，其他大输出命令（`log`、`diff`、`grep`、`ls-files`、`show`、`for-each-ref`）仍可能在 `println!` 上 panic。
- **范围**：全局输出层 + 上述大输出命令。
- **精确验收**：任一大输出命令在下游提前关闭时不打印 panic/backtrace；`EPIPE` 视为正常终止；退出行为对齐常见 Unix 工具（进程收到 SIGPIPE 语义或以约定码静默退出）。建议在 `main.rs` 建立统一 EPIPE 收口（写失败判 `BrokenPipe` → 静默退出），而非逐命令打补丁。
- **执行决策（2026-07-09）**：保持既有 `Pager`/`cat-file`/`rev_list_output` 局部处理，同时新增全局 stdout broken-pipe 收口：`main.rs` 安装 panic hook，识别 Rust `println!`/`print!` 在 stdout 关闭后的特定 panic 并静默退出；`utils::output` 把 JSON/human emit 的 stdout `BrokenPipe` 映射为 `silent_exit(0)`；`ls-files` 的直接 stdout 写入改走同一 helper。测试用直接关闭子进程 stdout 读端来稳定模拟 `| head` 提前退出，并覆盖 `log`、`diff`、`grep`、`ls-files`、`show`、`for-each-ref`、`cat-file` 和 JSON emit。
- **测试**：`tests/compat/broken_pipe_output_test.rs`；cargo target `compat_broken_pipe_output`。

### P0-07 修复 `commit --amend --no-edit` 的 no-op 成功问题

- **问题**：clean amend 打印成功但 HEAD 不变，脚本误以为刷新了 committer metadata/signature/hooks。
- **证据**：`§11`（空 `commit --amend --no-edit` 行为不同）。
- **范围**：`commit --amend`。
- **精确验收**：Git 兼容重写 commit（即使 tree 不变也刷新 committer/date/signature 并生成新 OID）；若产品决定在“无任何可重写内容”时 no-op，则**不得打印真实提交摘要**，须打印显式的“nothing to amend / HEAD unchanged”并保持 HEAD 与退出码可区分。
- **执行决策（2026-07-09）**：选择 Git 兼容的“总是重写”路径，不采用 no-op 文案分支。`commit --amend --no-edit` 在 clean 场景下复用父提交 message/author/tree/parents；如果新提交除 committer timestamp 外与父提交一致且 timestamp 未推进，则强制把 committer timestamp 设为父提交之后，保证写出新 OID 并移动 `HEAD`。签名路径在 timestamp 修正后执行，避免签名内容与最终 commit 不一致。

### P0-11 修复 symlink 基础兼容

- **问题**：status/add 忽略 symlink，checkout 可能把 tracked symlink 写为普通文件（内容为目标路径字符串）。
- **证据**：`§934`（`status`/`add` 忽略文件系统 symlink）。
- **范围**：index mode `120000`、checkout/restore/reset、status、add/update-index、diff、ls-files。
- **安全要求（新增）**：从 tree 数据物化 symlink 时防路径逃逸/TOCTOU —— 不得在 checkout/restore/reset 或冲突标记重建期间跟随指向 worktree 之外的 symlink 写入，link 内容按字节还原而非解析。
- **精确验收**：Unix 下 `add` 一个 symlink → index mode `120000`、blob 内容为链接目标；`checkout`/`restore`/`reset --hard` 还原为真实 symlink；`status`/`diff`/`ls-files -d/-m` 正确识别 symlink 变更且 dangling symlink 不被误判删除；`reset <target> -- <path>` 保留 `120000`；不支持平台（Windows 无符号链接权限）必须显式 skip/diagnostic，不静默写成普通文件。
- **执行决策（2026-07-09，已验收 v0.18.45）**：源码核对发现 `update-index` 已有部分 mode `120000` 支持但 link target 读取有 lossy/跟随风险，`add`/`diff` 会通过普通文件读取跟随 symlink，`status`/untracked walker 与 `ls-files -d/-m` 使用 `exists()`/`is_file()` 漏掉 dangling symlink，`restore`/`reset --hard` 会把 tree 中 symlink 写成普通文件，`reset <target> -- <path>` 写回 index 时会丢失 symlink mode，`restore --ours/--theirs` 曾只按 stage blob hash 写普通文件。当前实现统一用 `symlink_metadata` 识别路径本身，symlink blob 内容取 link target bytes；restore/checkout/reset 按 `TreeItemMode::Link` 在 Unix 创建真实 symlink，普通文件与 `restore --merge` 冲突标记写入前移除同名 symlink；pathspec reset 保留 tree item mode；非支持平台显式返回 unsupported 诊断。Codex review 追加修复了 `ls-files -d/-m` dangling symlink、reset pathspec mode、conflict stage mode、`add --renormalize --dry-run` symlink 预览以及 `restore --merge` 写穿 symlink 的风险。文档同步 `docs/commands/{add,checkout,restore,reset,status,diff,ls-files,update-index}.md` 及 zh-CN、对应 `docs/development/commands/*`、`COMPATIBILITY.md`、`tests/INDEX.md` 与 `tests/compat/README.md`。
- **测试**：`tests/compat/symlink_basic_test.rs`（Unix 门控；Windows 断言显式降级）。验收命令：`cargo +nightly fmt --all --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`source .env.test && cargo test --test compat_symlink_basic`、`source .env.test && cargo test --test compat_command_docs_examples_section`、`source .env.test && cargo test --test compat_matrix_alignment`。

- **问题**：全局 config 库（`~/.libra/config.db`）schema 版本比当前二进制支持的新时，`libra pull` 等命令**静默忽略**全局存储配置并退回 local storage，可能导致 pull 源/远端存储不是用户预期的，且用户无从察觉。

### P1-01 共享 Git pathspec engine

- **落地子步骤（增量，GC-05）**：

### P1-04 补齐 pretty-format renderer

- **目标**：`log`/`show`/`shortlog`/`for-each-ref` 不再把常用 placeholders 原样打印。

### IX-01 安装时创建可选 `lba` 别名（Issue #437）

- **目标**：安装时在 `libra` 旁创建短别名 `lba`，降低日常输入成本，不改项目名。
- **证据**：Issue #437（`~/.libra/bin/lba -> libra`，可自动或由 installer flag 控制）。
- **现状核实**：`install.sh` 当前参数解析在 `parse_args()`，支持 `--no-modify-path` 但无 alias 选项；二进制安装在 `screen_install()` 中 `mv "$temp_file" "$target"`（`target=$INSTALL_DIR/libra`）；同版本已安装会在 `main()` 中走 `screen_already_installed` 短路。二进制经 clap 分发，与 `argv[0]` 无关，故 symlink 即可直接工作，无需改 Rust 代码。
- **实现（`install.sh`，紧随二进制安装步骤之后）**：
  - 在 `parse_args()` 增加 `--no-alias`，并在 `usage()`/环境变量说明中登记；新增 `CREATE_ALIAS=1` 默认值，`--no-alias` 或 `LIBRA_NO_ALIAS=1` 设为 0。
  - 抽一个 `ensure_lba_alias()` helper，在 `screen_install()` 成功安装 `libra` 后调用；**同版本已安装短路前也要调用**，这样重复运行安装器能修复缺失的 `lba`。
  - 默认创建 `"$INSTALL_DIR/lba"` → `libra` 的相对 symlink（`ln -sfn libra "$INSTALL_DIR/lba"` 或等价安全实现，用相对目标以便整目录搬迁）。
  - 幂等：重复安装不报错；已存在且指向 `libra` 或 `"$INSTALL_DIR/libra"` 的 symlink 直接覆盖刷新。
  - **不覆盖非 libra 的既有 `lba`**：若 `$INSTALL_DIR/lba` 已存在且不是指向 `libra`/`$INSTALL_DIR/libra` 的 symlink（可能是用户自有可执行文件），跳过并 `warn_fact` 提示，不 clobber；不得 `rm -f` 后再判断。
  - 平台回退：不支持 symlink 的平台（Windows 无权限/非 symlink FS）改为跳过并提示，或按平台惯例创建 `lba.cmd`/`lba.ps1` shim（可作为后续子项，Unix 优先）。
  - 卸载/清理：若安装脚本含 uninstall 或 EXISTING_PATH 清理路径，一并处理 `lba`。
  - success box / `agent_say` 提示用户新可用 `lba`；`screen_already_installed` 也要提示 alias 是否已创建/跳过。
- **文档**：README 安装段与安装文档说明 `lba` 别名及 opt-out。
- **验收**：干净安装后 `lba` 存在且解析到同一 `libra`（`lba --version` == `libra --version`）；同版本重复运行会补建缺失 alias；`--no-alias` 与 `LIBRA_NO_ALIAS=1` 均不创建；已存在的非 libra `lba` 不被覆盖；已存在合法 alias 重跑幂等；`shellcheck install.sh` 无新增告警。
- **测试**：`install.sh` 无内建测试框架，采用 `shellcheck` + 一段可在 CI/本地跑的 POSIX 冒烟脚本（临时 `INSTALL_DIR`，跳过真实下载、用桩 `libra`）验证上述断言。若直接跑完整 installer 成本过高，先把 alias 逻辑抽成可 source 的 helper 或新增 `LIBRA_INSTALLER_TEST_MODE=alias`，但测试必须覆盖同版本短路补建路径。

## 验收门禁

- **每个任务必须有对应 compat/integration test，不接受只更新文档。**
- **每个改动同步**：`COMPATIBILITY.md`、对应命令文档（EN↔zh parity，GC-10）、`docs/development/commands/_compatibility.md`（unsupported 项须有 D 编号或本计划编号）。
- **每个命令改动同步用户文档**：更新 `docs/commands/<cmd>.md` 和 `docs/commands/zh-CN/<cmd>.md`；若不存在则创建 EN/zh-CN 文档，并补 examples/JSON/error-code 说明。
- **涉及用户可见错误**：使用稳定 `LBR-*` 错误码并同步 `docs/error-codes.md`（新增如 `LBR-CONFIG-001`）。
- **compat-surface 守卫（新命令/flag/错误码必查，否则 CI 红）**：`compat_matrix_alignment`、`compat_help_examples_banner` + `root_after_help_lists_every_visible_command` + `compat_command_docs_examples_section`（EXAMPLES 契约）、`compat_error_codes_doc_sync`；新增 `[[test]]` target 同步 `Cargo.toml` 与 `tests/INDEX.md`。
- **每项必跑三门（本地全绿方可 review/发布）**：`cargo +nightly fmt --all --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`source .env.test && cargo test <该改进指定测试用例>`；全量 `cargo test --all` 可作为阶段性加强，但不能替代每项指定测试。
- **Codex review 必须通过**：每项改进在本地验收后交由 Codex review；根据 review 结果修改并重跑验收，直到 review 通过。
- **Libra main 发布门禁**：每项改进完成后 patch 版本 +1，同步 `Cargo.toml`、`web/package.json`、`worker/package.json`，执行 `cargo build --release` 更新 `Cargo.lock` 和 release 产物，复制 `target/release/libra` 到 `$HOME/.libra/bin/libra`，再用 `libra add <相关路径>` / `libra commit -s -m`（**不加 `-a`**，见「执行检查必备需求」第 6 条）/ `libra push origin main` 提交推送；push 遇 origin 版本领先按「版本抢号与 push 冲突解决」处理；不创建 branch/worktree。
- **失败处理**：push 失败不重试，等待下一次修复完成后再推；其他内部服务暂时错误按 10 分钟 backoff 重试，不暂停任务目标。
- **测试环境注意**：macOS 上跑全量需 `RUST_MIN_STACK=33554432`（否则 tag 测试 SIGABRT 掩盖结果）；真机 Git 互操作用例在无 `git` 时 gate skip，不硬失败（GC-09）。

## 非目标提醒

- 不以 `libra_git.md` 的 ~940 项逐项清零作为成功标准。
- 不把 Git 老旧命令和辅助 helper 全量搬进 Libra。
- 不用“有意差异”掩盖静默错误：凡命令成功但状态、历史、对象、配置或机器输出误导用户，都必须修或降级承诺。
