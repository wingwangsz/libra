## 结论

**当前不能直接取消 TUI。** 第三轮源码复核（`code.rs`、`headless.rs`、`code_ui.rs`、`app.rs`、`session/jsonl.rs`、`run_admission.rs`、`runtime/hardening.rs`、`code.md`）支持本文主结论：Agent-first → Web 接入 → 关闭 TUI 的顺序正确；在「20260708 已全部实现」前提下，最大执行风险是 **A0 接口漂移**（含 A0-05 fix-bridge 依赖方向反转，F37）、**三态并行 owner 未收敛**、**code.md 与 Web-default 契约冲突**，而非任务拆分本身。

当前实现已经具备三路入口：默认 TUI、`--web`/`--web-only` headless Web、`--stdio` MCP transport（`src/command/code.rs:696-704`）。其中 `--web-only` 已能把非 Codex provider 路由到 `HeadlessCodeRuntime`，并能通过浏览器提交 direct turn（`src/command/code.rs:795-868`，`src/internal/ai/web/headless.rs:290-375`）。但是这条 Web-only 路径不是默认 TUI 的功能全集：

- `docs/development/tracing/code.md` 仍把默认模式定义为“交互式 TUI + 后台服务”，普通请求先走 IntentSpec / 执行计划审阅确认（`docs/development/tracing/code.md:9-12`）。
- Web-only headless runtime 源码明确说 Phase 0/1 IntentSpec / Plan workflow 仍是后续目标，当前每次 browser submit 被当成单次 direct turn（`src/internal/ai/web/headless.rs:21-28`）。
- 非 TUI mode 仍拒绝 `--resume`、`--env-file`、`--context`、`--approval-policy`、`--approval-ttl`、`--network-access allow`（`src/command/code.rs:4124-4130`，`src/command/code.rs:4335-4398`）。
- TUI 仍拥有 turn id、snapshot mutation、interaction lifecycle；Web/automation 写入在默认模式下通过 `TuiCodeUiAdapter` 转发到 App main loop（`src/internal/tui/code_ui_adapter.rs:1-7`，`src/internal/tui/app.rs:1063-1103`）。
- TUI 仍承载 Phase 0/1 plan workflow、post-plan/network policy、goal/task control 等关键行为（`src/internal/tui/app.rs:6995-7068`，`src/internal/tui/app.rs:5884-5955`，`src/internal/tui/app.rs:6377-6445`，`src/internal/tui/control.rs:38-75`）。
- `CodeUiCommandAdapter` 对非 TUI adapter 默认不支持 `task.dispatch` 与 Goal mode（`src/internal/ai/web/code_ui.rs:743-760`）。

因此，**Web-only 是最终可行方向，但实现顺序必须调整为：先把 Agent 功能独立成模块，再由 Web 接入该模块完成工作流，最后关闭 TUI**。目标状态是：`libra code` 默认启动 Web Code UI + 独立 AgentRuntime；Web/API 是唯一交互写入面；TUI 不再拥有 agent/session 行为，也不再作为生产控制面；独立 `libra code-control` 命令取消，原 `code-control --stdio` 能力迁入 `libra code --control stdio`（用户口径中的 `libra code -control` 若需要兼容，只能作为同一字段的解析别名，不能成为第二套命令）。

## 调整后的实现顺序

本文后续任务按以下依赖顺序执行，不能倒置：

1. **Agent 模块先独立**：把 `libra code` 的 provider/tool/approval/sandbox/session/goal/task/plan workflow 等能力从 `src/internal/tui/app.rs` 和 headless Web adapter 中抽离，收敛到 UI-neutral `AgentRuntime` / `CodeAgentServices` 模块。按“20260708 已全部实现”的前提，本步应先复用 A0 已落地的 queue / approval / sandbox / tracing / trust / skill projection 原语（五轮实况见 F36：trust/skill projection 尚未落地，须按 W0-01 漂移流程处理；mutating 安全边界即 hardening + tool loop，A0-05 bridge 为 deferred，F37）；只有 Code 会话专有的 turn worker、Code UI projection、browser/control adapter 可以在本文新增。完成标准是该模块可以独立驱动一次受控编码会话，不依赖 TUI event loop。
2. **Web 再接入 Agent 模块**：Web Code UI、HTTP/SSE、browser control、`libra code --control stdio` 自动化客户端只调用 AgentRuntime handle；Web 负责展示、输入、交互响应和观察，不拥有 plan workflow、Goal、usage、session projection 私有状态。
3. **最后关闭 TUI**：只有当 Web 端已覆盖默认编码工作流、resume、approval/cancel、Goal/task、Codex/non-Codex provider、SSE/security 之后，才切默认入口并删除 TUI startup、TUI control bridge、Code PTY harness 和 graph TUI。

### 当前三态所有权（数据流基线）

迁移前 `libra code` 存在**三个并行写/投影 owner**，是本文禁止「双实现」的根因：

```text
TUI 默认路径
  execute() -> execute_tui()
  App (app.rs) 拥有 IntentSpec/Plan/approval/goal/task 状态机
  TuiCodeUiAdapter 把 HTTP 写入转成 TuiControlCommand -> App main loop
  SessionStore + metadata 内嵌 CodeUiSessionSnapshot（headless 兼容键）

--web-only 非 Codex
  execute() -> execute_web_only() -> HeadlessCodeRuntime
  每次 browser submit = 单次 direct turn（跳过 Phase 0/1 审批环）
  CodeEnvFile::default()；--resume CLI 被拒绝

--web-only Codex
  execute() -> managed app-server + CodexCodeUiAdapter
  approval/user-input 可能回环外部 app-server；与 headless 投影形状需归一（W3-04）
```

目标态收敛为：**单一 `AgentRuntimeWorker` owner** + Web/API/`--control stdio` adapter 只读投影/只经 API 写入。

## Web-only 决策

### 允许的目标

- 可以取消 `libra code` 的 TUI 生产入口。
- 可以把 `libra code` 收敛为 Web Code UI + AgentRuntime。
- 可以删除 `TuiCodeUiAdapter`、`TuiControlCommand`、Code TUI startup、Code 专用 PTY harness、独立 graph TUI。
- 可以把 `--web` / `--web-only` 从“选择 Web”变成冗余 alias，最终拒绝或删除。
- 可以取消独立 `libra code-control` 命令，把原 `code-control --stdio` 的自动化客户端能力迁入 `libra code --control stdio`；`--control observe|write` 作为启动 Web Code UI 的控制模式继续保留。

### 不允许的捷径

- 不允许先改 `execute()` 默认分支为 Web-only direct-turn，再把 IntentSpec / Plan 审批标成后续。
- 不允许让 Web adapter 自己维护 plan workflow、Goal、usage、skill、session projection 的第二套状态机。
- 不允许把 MCP stdio 或 `libra code --control stdio` 当作 Agent turn 控制面绕过 token/lease/approval。
- 不允许在 `libra code-control` 被取消后继续保留公开命令文档、README 索引或 compat 承诺；旧命令只能作为 breaking-change 迁移说明出现。
- 不允许删除 TUI 测试前降低 approval/cancel/controller/resume/plan workflow 的覆盖。

## 总体架构目标

迁移后的生产结构按“Agent 模块为核心，Web 为 adapter”的方向收敛：

```text
libra code
  -> start Web Code UI
  -> start AgentRuntimeWorker per session
  -> Web/API adapter calls AgentRuntimeHandle
  -> AgentRuntime owns:
       serialized turn queue
       IntentSpec / Plan / approval / user-input interaction state
       provider stream normalization
       tool loop, sandbox, approval, hooks
       goal, skill, usage, task command service
       append-only SessionEvent log
       snapshot/projection fold
       cancel/resume/recovery
```

TUI 在迁移期只作为行为冻结参照和临时观察面；完成后不得再拥有 agent/session 状态，也不得作为 Web write path 的中转。

独立 Agent 模块的最小边界：

- `AgentRuntimeHandle`：Web/API/`libra code --control stdio` automation 调用的 submit/respond/cancel/observe/snapshot 入口。
- `AgentRuntimeWorker`：单 session serialized queue 与 turn/interaction 状态机。
- `CodeAgentServices`：provider、tool registry、approval/sandbox、hooks、skills、usage、MCP bridge、session store 的统一装配。
- `AgentEvent` / `SessionEvent`：Web snapshot、SSE、resume、graph projection 的唯一事实源。
