## 文档职责

本文参照 `entireio/cli`（Entire CLI，Go 实现，仓库 <https://github.com/entireio/cli>）的**外部 agent 会话数据读取方案**，评估 Libra 当前对 Claude Code / Codex / OpenCode 三类外部 agent 的数据获取能力，并把差距收敛成可落地的改进计划。

核心判断（贯穿全文，**双源模型**）：外部 agent 捕获有两个不可互相替代的数据源——

- **transcript = 权威内容源**：完整对话、token 用量、模型、修改文件、子 agent 等，都从 agent 落盘的 session/transcript 文件解析（`runtime.rs` 读原始 transcript → 再 redact）。
- **hook = 生命周期 / 归属 / checkpoint 边界源**：hook 不只是「递指针」——它还负责**创建/更新 session、提供 cwd 与 metadata、界定 checkpoint 生成时机**。当前 committed checkpoint 只在 `TurnEnd`(Stop)/`SessionEnd`/subagent 边界物化；没有这些 hook 生命周期事件，writer 根本不会跑。

因此本文主张的**不是**「注册更多 hook 事件」，而是「**补齐 transcript 内容源的获取入口**」。`entireio/cli` 只注册 6 个 Claude hook 事件却能拿到完整数据，正因为它把内容读取做厚了。Libra 的 transcript 解析层（尤其 Claude）已很完整，但内容获取入口几乎只有「hook 直接递来的那一个 `transcript_path`」，缺少 Entire 已具备的**独立发现、批量导入、异步落盘等待、OpenCode 导出桥接**四类入口。批量导入（DR-05）因此必须**自行合成 hook 侧的生命周期语义**（cwd / 起止时间 / turn 边界 / session 状态），不能假装「复用 hook 管线」。

本文只规划任务，不宣称已完成。落地时每个任务先刷新源码锚点，再按任务卡验收；文中行号是 2026-07-13 复核的证据快照，不是永久事实源。外部 Entire 参照也必须在任务卡中记录**精确 commit SHA、探针版本和已核对文件**；不得把其浮动 `main` 分支当作可复现规范。

### 非阻断但须知的残差风险

- 每 TurnEnd 全量 read/redact/hash 是**既有基线**，本计划不治理；长会话仍 O(n²) CPU/IO。
- **live blob 保留 generic-redacted unknown fields（接受的隐私残差）**：live checkpoint 的整份 transcript blob 含未识别 provider 字段的脱敏后原文（ADR-DR-12 兼容例外）。缓解：generic `Redactor` 覆盖已知密钥模式后才落盘；blob 只进本仓库 traces 与用户显式的 `agent push`。未被规则覆盖的新型敏感字段仍可能以明文残留——该风险为既有基线的一部分，接受并记账；消除手段（live allowlist 投影）另立 RFC。
- OpenCode `session.deleted` 在 headless 未观测到 → 终态依赖 idle；进程被杀且无后续 idle/import 时最后若干 turn 可能延迟可见（best-effort）。
- Claude project-slug 转义规则若上游变更，DR-02 发现会漏文件；应用探针版本钉死 + doctor 提示。
- 现有 cloud capture mirror 尚无 delete/tombstone propagation；本文新增的 import tombstone 只阻止本地 discovery/import 自动复活，**不宣称**解决 cloud restore 复活，继续沿用 A0-10 的 deferred 限制。故本功能不能作为跨设备合规擦除或保留策略的唯一控制。

### ADR-DR-02 — 统一 TranscriptSource seam（取代双轨 preparer）

- **唯一 writer 读入口**：`resolve_transcript_source(ctx) -> Result<Option<TranscriptSource>>`，其中  
  `TranscriptSource = File { file: AuthorizedTranscriptFile, source_id, auth: ProviderRootAuthorized } | Bytes { bytes, auth: ExportAuthorized { agent_kind, session_id, content_digest } }`。`AuthorizedTranscriptFile` 表示已经按 ADR-DR-13 安全打开并完成 beneath/no-follow 校验的 handle；writer 不得重新按 path 打开。
- **既有** `TranscriptPreparer::prepare_transcript -> ()`：
  - 可保留为可选 **副作用钩子**（flush-wait），由 `resolve_transcript_source` 在安全打开并返回 File 前调用；
  - **不得**再作为第二条「能改写路径/字节」的公共 API；新代码只认 seam。
- **信任校验**：
  - `File`：provider-root resolver 负责安全打开并返回授权 handle；现有 `transcript_path_within_provider_root` 只能作为迁移期预检，不能作为最终 TOCTOU 安全边界。
  - `Bytes`：**禁止**伪造「已授权」；仅 OpenCode export 桥接（及未来同类 bridge）在 sandbox 成功后构造；writer 校验 `auth` 标签与 session 绑定，不把任意 `&[u8]` 当可信源。
- 调用点：`write_committed_checkpoint` / import writer 共用；禁止 import 旁路 seam 直接 `fs::read` 或授权后重新按路径打开。
- **`AgentSessionCtx` 演进**：`transcript_path: Option<PathBuf>` 仅保留为 resolver 输入/兼容字段，writer 不再直接读它；新代码只消费 seam 返回的 handle/bytes。若 OpenCode 等 Bytes 分支需要额外上下文（如 exporter argv、generation），可扩展 `AgentSessionCtx` 或传入独立 bridge 参数，禁止把 bytes 直接塞进 `transcript_path` 字段冒充路径。

### ADR-DR-03 — 安全与 redaction 顺序

安全顺序以 ADR-DR-12 为唯一权威：获取源 → 有界不可信解析 → typed normalize → typed-field redact → canonicalize/digest → claim → persist。  
**禁止**未 redact 的原文进入 SQLite blob、对象库、日志或默认磁盘缓存；不得再保留“先整段 redact 后 split”和“必要时先 split”两套行为。
OpenCode：默认 **不落 export 原文缓存**；若调试需要落盘，必须专用私有目录 + `0600` + `O_NOFOLLOW` + TTL/退出清理，且不得进入 sync 路径。

### ADR-DR-04 — live 与 import 的 checkpoint 粒度

| 路径 | 默认行为 | coverage |
|---|---|---|
| live `TurnEnd`/`SessionEnd` | 保持既有：事件触发时写 checkpoint；blob 仍为**当时可见的整份** redacted transcript（既有基线） | 对该事件负责的逻辑 turn claim；若同一 turn 已存在等价/更完整可见修订则 no-op；若现有修订为 `incomplete` 且本次为 `complete`，写新的 **claim revision**，不在 checkpoint 级隐藏旧快照 |
| import 默认 | **按 turn 拆分**写入，与 live 逻辑键对齐；每个未覆盖或可合法升级的 turn 一个 checkpoint | 与 live 同一 `(session_id, logical_turn_key, coverage_schema_version)`；digest 仅比较内容版本，不参与逻辑唯一键 |
| import `--single-session-checkpoint` | **移出本计划** | 部分 turn 已覆盖时语义不唯一；未来若重启，必须采用“全部 turn 均可一次原子预留，否则整次拒绝”的 all-or-nothing RFC |

- live 每次解析当前快照的全部 normalized turns，并计算“本事件尚未覆盖/可升级”的 turn 集合；一个 whole-transcript checkpoint 可以被多条 claim 指向，所有 claims 与该 checkpoint 在同一最终事务提交。不能只假设永远恰好一个“刚完成 turn”。**因此 checkpoint 不是 turn revision 的所有者：任一 claim 升级都不得设置该共享 checkpoint 的 `superseded_by` 或把它从默认 checkpoint list 隐藏。**
- import per-turn checkpoint 的 transcript blob 是从 typed normalized 记录**重新序列化的 provider-tagged 安全投影**，不是「原始 JSON 只改几个字段」。仅 allowlist 的结构与字段可保留，未知字段一律丢弃；canonical bytes 仅用于 coverage digest，不替换用户可见的安全投影，也不制造与 live 不同的事实语义。

### ADR-DR-05 — 子 agent（DR-06）对齐 A0-02

- **A0-02 已完成态（按源码修正）**：hook 路径 `SubagentStart`/`SubagentEnd`（`LifecycleEventKind`）已物化独立 `scope='subagent'` **boundary checkpoint**，transcript 为空；当前 provider envelope 没有可靠的子 agent ID/tool-use ID，实际 `subagent_session_id` / `tool_use_id` 均硬编码为 `None`（`runtime.rs:1294` 附近），不能假设它能与磁盘文件一一关联。
- **boundary 来源限定（关键前提）**：这些 `SubagentStart`/`SubagentEnd` 生命周期事件当前**只由 Codex 原生子 agent hook**（`hooks/providers/codex/`，转发为 `subagent-start`/`subagent-end`）产生；**Claude 的 5 事件 forward map（`CLAUDE_HOOK_FORWARD_MAP`：SessionStart/UserPromptSubmit/PostToolUse/Stop/SessionEnd）不含任何 subagent 边界事件**。因此 DR-06 面向的 **Claude `<sessionID>/subagents/` 磁盘内容默认没有可关联的 A0-02 boundary**，其 content checkpoint 常态即 `link_state='unresolved'`；本计划不得承诺“Claude 子 agent content 与 boundary 自动关联”，除非 Claude 未来也发出稳定的 subagent 边界信号。
- 磁盘 `<sessionID>/subagents/*.jsonl`：每个子文件 → 独立 `scope='subagent'` **content checkpoint**；稳定身份使用 `(parent_session_id, provider_kind, provider-root-relative source key, content_schema_version)`，不得用 basename 单独做键，也不得伪造 hook 未提供的 ID。
  - boundary 与 content 是两类不同证据：默认均保留；通过可空 `boundary_checkpoint_id` 或关联表表达“best-effort 关联”，不 update-in-place、不重写不可变 traces 历史。
  - 只有在 provider 明确给出稳定 subagent/tool-use ID 且匹配唯一时才自动关联；否则标 `link_state='unresolved'`，由 doctor 报告，不猜测。
  - 同一 content source 的重复发现由逻辑 source key + content digest 幂等；内容版本使用 source-scoped revision/当前叶关系表达，**不得**复用可能被多个 turn 共用的 `agent_checkpoint.superseded_by`。物理历史允许旧修订存在，但默认最多一个当前 content 叶。
- `SubagentAwareExtractor`：可扩展为多源输入，用于父 checkpoint metadata 的 **并集汇总**（modified files 并集、usage 可加总或标 partial）；**不得**再把 `subagent_usage = 主 session 总量` 当作「已归属」。
- 与 hook 路径并存时，不把 boundary 当 content 的重复项；分别幂等，并保证 list/export 能区分 `boundary` 与 `content` provenance。

### ADR-DR-06 — 两表分层 + 关系约束

| 表 | 职责 | 谁写 |
|---|---|---|
| `agent_coverage_claim` | **写前并发门 + 当前覆盖事实**：某 session 的某逻辑 turn 当前可见修订及写入状态 | live + import |
| `agent_coverage_revision` | **不可变 turn 修订史**：每次已提交 coverage 的 digest/completeness/checkpoint，表达 per-turn 的 supersede 关系 | live + import |
| `agent_import_identity` | **import 作业状态**：稳定身份、分阶段状态、lease/fence、last/attempt checkpoint | **仅 import** |
| `agent_export_job` | OpenCode export 在途 observed/processed generation、owner/fence、dirty 状态 | **仅 OpenCode export 路径** |
| `agent_import_tombstone` | 用户擦除后的本地防复活标记 | erase + discovery/import 读取 |

live **不**写 `agent_import_identity`。跨路径去重只看 coverage 门键。

**关系与清理**：
- `agent_coverage_claim.session_id` 外键指向 `agent_session.session_id`，`ON DELETE CASCADE`；claim 行随 session erase 自动清理。
- `agent_coverage_claim.checkpoint_id` 可空，最终指向 `agent_checkpoint.checkpoint_id`；`agent_coverage_revision` 使用 `(session_id, logical_turn_key, coverage_schema_version, revision)` 作为不可变主键，并记录 `checkpoint_id`、digest、completeness、source_channel、created_at。`session_id` 外键同样 `ON DELETE CASCADE`，`checkpoint_id` 在最终事务中验证为已有 catalog 行；最终事务同时 INSERT revision 并将 claim 前推至该 revision。这样一个 checkpoint 可安全服务多条 claim，修订关系永不借用 `agent_checkpoint.parent_checkpoint_id` 或 checkpoint 级 `superseded_by`。
- ref/catalog/claim/revision 最终提交在同一 SQLite 事务内，不以“避免死锁”为由放弃原子性。若 SQLite 外键约束与对象构造顺序冲突，允许 deferred logical validation，但 `state='catalog_committed'` 前必须验证 catalog 行存在且 traces commit 可达。
- `agent_import_identity` 独立存在，稳定键不得直接使用会泄露 home 的绝对路径；优先 `(agent_kind, provider_session_id, source_kind, provider-root-relative source_id, schema_version)`，不存在稳定 provider ID 时再使用加盐 source fingerprint。session 删除时清理运行状态，但 erase 必须先写 tombstone。
- `agent_export_job` 列：`job_id`、`agent_kind`、`provider_session_id`、`owner`、`lease_expires_at`、`fence_token`、`observed_generation`、`processed_generation`、`state(idle|inflight|dirty|failed)`、`last_error_code`、`created_at`、`updated_at`、`ttl_expires_at`；按 `(agent_kind, provider_session_id)` 唯一。
- `agent_import_tombstone` 的**导入阻断键**是 `(agent_kind, provider_session_id)` UNIQUE；另须保存不带外键的 `erased_session_id TEXT NOT NULL UNIQUE`，erase 在删除 `agent_session` 前**同时写入这两种身份**。`libra agent graph <session>`（DR-07）只按 `erased_session_id` 只读查询 tombstone（不作为导入匹配条件），因此 catalog 行已删除后仍能把 `<session_id>` 稳定判为 `erased` 而非 `unknown`。`source_fingerprint` 仅作不可反向的审计属性，不能成为匹配条件，否则同一 session 被移动/复制后会复活。普通 retention 不删除。显式 `--restore-erased` 才可移除/绕过，且 machine/non-TTY 默认拒绝未确认操作。该表默认仅本地，不宣称已有 cloud tombstone propagation。

### ADR-DR-07 — DR-01 策略择一

- **选定**：Claude Stop/TurnEnd 前 **有界同步 flush-wait**（默认预算 ≤2s，且整条 hook 关键路径须 < hook timeout 10s，为大 transcript 的 read/redact/write **留余量**；stale mtime 跳过；超时则尽力读并将末 turn 标为 `incomplete` + warn）。
- **非目标（本计划不做）**：provisional checkpoint + 后台 DR-05 回填双轨。若未来要做，单独立项，避免与 claim 状态机缠绕。

### ADR-DR-08 — coverage 门键与 canonical 形式

`agent_coverage_claim` 的逻辑键 `(session_id, logical_turn_key, coverage_schema_version)` 必须在 live、import、export 三条路径上产生一致值。`coverage_digest` 是该逻辑 turn 的内容版本，不再进入 UNIQUE；否则截断版和完整版会被错误识别为两个 turn。

1. **`coverage_schema_version`**：从 1 开始；覆盖规范变更（turn 分组规则、canonical JSON 格式、字段 allowlist）时递增。所有路径必须读同版本规范；旧版本 claim 对应当前版本视为未覆盖，不得静默复用。
2. **`logical_turn_key`**：provider parser 输出的稳定逻辑身份：优先使用经过格式校验的 provider turn/message ID；没有稳定 ID 时用 `normalized turn ordinal`。不得使用原始 JSONL 行号，因为插入 metadata/system 行会整体漂移。ordinal 模式必须验证旧 turn canonical 前缀未重排；发生重排进入 `conflicted`，不得静默当新 turn。
3. **`coverage_digest`**：单个 normalized turn 规范记录的 sha256(canonical bytes)。
   - **per-turn 覆盖**：集合只含该 turn 的 user、assistant、tool call、tool result/tool error 规范记录。
   - **live whole-transcript checkpoint**：blob 仍是整份 transcript，但 claims 只覆盖本次快照中尚未覆盖或可升级的 turn 集合；每条 digest 分别按对应 turn 算，**不能**按整 blob 计算。
4. **内容完整性**：claim 增加 `completeness(incomplete|complete)` 与 `revision`，每个已提交版本写入 `agent_coverage_revision`。flush timeout、损坏尾行、export 截断只能产生 `incomplete`；后续 complete 内容可原子前推该 turn 的 current revision。complete→不同 complete 不自动覆盖，进入 conflict/doctor。该关系仅影响该 turn 的查询/UI，不得隐藏仍为其他 turn 提供证据的 whole-transcript checkpoint。
5. **canonical 协议**：单独新增 `docs/development/tracing/coverage-v1.md`，采用 RFC 8785 JCS 或逐项等价的项目规范；必须定义 UTF-8/Unicode、对象递归键排序、数字、escape、duplicate-key 拒绝、缺失与 null、tool input/result/error、文本块顺序及未知字段策略。固定 golden vectors 必须给出输入、canonical bytes 和 sha256。
6. **source identity**：持久化 root-relative source ID 或摘要；绝对 canonical path 只允许存在于本地短时诊断内存/受权限保护的本地表，不进入 syncable metadata。

### ADR-DR-09 — live 与 import 写冲突仲裁

- **权威定义收窄且可实现**：同一逻辑 turn 的等价 complete 内容，先完成原子提交者可复用；live 可抢占尚未提交 ref 的 import reservation。若 import 已提交 complete 等价内容，live no-op，不为了 provenance 重写。若内容不同，按 incomplete→complete 升级或 conflict 规则处理，不以 source channel 粗暴覆盖。
- **实现规则**：
  1. 写对象前获取/创建 `state=reserved_{live|import}` claim，记录 owner、lease、fence、attempt checkpoint；逻辑键冲突后按 state/completeness/digest 决定 no-op、takeover、upgrade 或 conflict。
  2. live 抢占 `reserved_import` 必须条件 UPDATE owner/source/state，并递增 `fence_token`；陈旧 import 后续任何 ref、claim、identity writeback 都必须失败。
  3. import 已 `catalog_committed` 且 digest 等价时，live 复用其 checkpoint；文档不再宣称“live 无条件胜出”。
  4. 并发竞争通过 `rows_affected = 1` 判定；失败方重新读状态后决策，不允许直接假设 no-op。
- **推论**：`source_channel` 仅用于 provenance 与诊断，不参与去重胜负。

### ADR-DR-10 — 原子提交、状态机与 DB 故障策略

- claim 状态固定为 `reserved_live|reserved_import|catalog_committed|abandoned|conflicted`；禁止用两态模型表达全部窗口。`superseded` 是 **claim revision** 关系，不是当前唯一 claim 行的终态；最终原子事务不暴露中间 `ref_committed` 状态。
- checkpoint 对象/tree/commit 可在 SQLite 事务外构造；最终事务必须一次完成：验证 owner+fence+expected state 与 tombstone 未命中 → 验证 traces ref expected head → 更新 ref → INSERT catalog → INSERT coverage revision → 更新 claim 为 `catalog_committed` 并挂 checkpoint/traces commit → 前推 import identity（若有）。
- `HistoryManager::update_ref_if_matches` 必须提供 transaction-injected 变体；若 helper 内部另开事务，则不满足本 ADR。
- **CAS 重建循环**：对象构造以读到的 `expected_traces_head` 为父；最终事务中 ref 条件更新为 0 行时必须回滚该事务、重新读取 head、在事务外以新 head **重建 tree/commit** 后再试。不得把基于旧 head 的 commit 直接接到新 head，也不得在重试中丢弃 reservation/fence/tombstone 检查。重试受 lease、总 deadline 与 GC-DR-08a 预算约束；超时只留下不可达对象并返回可重放失败。
- 多 turn import 的 claim 统一按 `logical_turn_key` 排序获取；涉及 identity 时固定锁/更新顺序为 identity → claims，避免锁序反转。
- DB/schema/claim 门不可用时 checkpoint 写入 **fail-closed**：不 append traces，只保留可重放的原 hook/import 请求或返回可操作错误。不得继续生成随机 UUID checkpoint，再依赖 metadata 事后去重。
- 状态不变量：
  - `catalog_committed` ⇒ checkpoint_id 非空、catalog 行存在且 traces commit 可达；
  - fence 被 takeover 后，旧 holder 最多留下不可达对象，不得更新 ref/claim/identity/job；
  - 每个 `(session_id, logical_turn_key, schema_version)` 最多一个 current、非 conflicted 的 **claim revision**；checkpoint 的可见性由其自身及所有引用它的 claim 决定，不以全局 superseded 位推断。

### ADR-DR-11 — OpenCode generation/lease 算法

- `agent_export_job` 使用 `observed_generation` 与 `processed_generation` 两个单调计数，不使用含义模糊的单一 `dirty_generation`。
- 每个 idle 原子 `observed_generation += 1`；能取得 lease 的调用者成为 runner，否则返回。runner 记录 target generation，export 并处理 coverage 后，在 owner+fence 条件下前推 `processed_generation`。
- 若处理完成时 `observed_generation > processed_generation`，在总 deadline/max loops 内继续；达到上限则保留 dirty 状态并释放/缩短 lease，等待下次 idle 或 takeover，不得误标 clean。
- stale runner 不能释放新 owner lease、覆盖 failed/idle 状态或前推 processed generation。job 清理由 TTL 索引驱动，并纳入 `agent clean --gc`、retention、doctor、session erase 与启动时有界 scavenging。

### ADR-DR-12 — 唯一 transcript 安全流水线

固定顺序为：bounded raw read/export pipe → **不可信结构解析（仅有界内存）** → typed normalized turns → 对 typed 字段逐字段 redact → canonical serialize/digest → claim → persist。解析 raw 不等于持久化 raw，不违反 redaction-before-persist。

- 原文字节不得进入 SQLite、对象库、日志、错误文本、默认缓存或 syncable metadata。
- 持久化 transcript 只能由 typed normalized model 的 allowlist 字段重新编码；unknown/extension/raw payload、未识别 attachment、任意 provider metadata、重复 JSON key 原文一律丢弃，不能以“原生片段”名义透传。错误对象同样只能携带 stable code、长度、摘要前缀和已脱敏诊断。
- **live-blob 兼容例外（由 ADR-DR-04 授予，M1 实现时明确化）**：live `TurnEnd`/`SessionEnd` checkpoint 的 transcript blob 保持**既有基线**——整份经 generic `Redactor` 脱敏的原始 JSONL（unknown fields 以脱敏后形态保留在 blob 中）。这不是本条的违例，而是 ADR-DR-04 表格里“blob 仍为当时可见的整份 redacted transcript（既有基线）”的既定语义，其存储代价与 unknown-field 保留已在「非阻断残差风险」与「性能与容量摘要」记账。**allowlist-only 重新编码约束的适用范围**：(a) coverage digest/claims——所有路径任何时候（typed normalize → typed-field redact → digest）；(b) import（M4）与 export（M3）的持久化 per-turn 投影。live blob 的 allowlist 化若未来要做，单独立项 RFC，不在本计划内。
- coverage v1 明确纳入 user/assistant/tool call/tool result/tool error 的语义内容；model、usage、时间戳、路径等 provenance 默认不进 digest，避免来源提取能力差异造成伪冲突。
- malformed/duplicate-key/超 cap 输入 fail-closed 或标 incomplete，不得在不同路径使用不同的“尽量解析”规则。

### ADR-DR-13 — `--path` 授权与 TOCTOU

- `--path` 始终只能读取对应 provider root 内的来源；`--yes` 只确认“将读取并脱敏写入本仓库”的隐私影响，**不扩大文件系统授权**。根外导入另立高风险 RFC，本计划不提供 `--allow-outside-provider-root`。
- 不能只做 `canonicalize + starts_with + 再按路径 read`。Linux 优先 fd-relative `openat2(RESOLVE_BENEATH|RESOLVE_NO_SYMLINKS|RESOLVE_NO_MAGICLINKS)`；其他平台使用逐组件安全打开/handle 校验，最终从已验证 handle 读取。

### ADR-DR-14 — 能力接口兼容

- `DeclaredAgentCaps` 的 8 个 bool 是已冻结公共 wire，本文不得增加、删除或重命名字段。
- `transcript_discoverable`、`importable`、`export_bridge` 通过 versioned `capabilities.methods[]` 或新的可选 extension object 暴露；新 consumer 不得按 agent slug 硬编码，并区分声明 `supported` 与环境实际 `available`/`unavailable_reason`。
- `libra agent list --json` 的默认 **v1 必须字段集、字段含义与 `schema_version: 1` 均保持不变**。新能力仅在显式 `--schema-version=2`（或等价明确协商 flag）请求时返回；v2 可含 methods/availability，v1 consumer 不会因未知字段或版本号变化失败。compat 测试必须同时 pin v1 字节形状与 v2 协商形状。

### ADR-DR-15 — erase、source 归属与防复活

- import 复用 `agent_session` 前必须校验 agent kind、provider session ID、规范化 working_dir/repository identity、provider root/source fingerprint；不一致返回稳定 conflict 错误，不静默绑定到另一项目。新发现的 session 也必须先证明其 working_dir 属于当前 repo identity；跨仓库会话不导入、不写 catalog，另立显式 RFC。
- `working_dir` 是既有 catalog 字段的兼容例外：import 不得持久化 transcript 中任意未经验证的 cwd，而必须写入本进程解析并已验证的当前 repo root。它不是 source transcript path；本计划不把既有 syncable working-dir 字段改为匿名标识，故不得声称解决该历史路径元数据的跨设备隐私问题。
- 本地 erase 在删除 session/claims/identity 前写 `agent_import_tombstone`；后续 discovery/import 默认跳过。显式恢复必须是可审计的高风险动作。
- 本 ADR 只处理本地自动 re-import；cloud mirror 的 delete/tombstone propagation 仍按 A0-10 deferred，文档必须继续提示 cloud restore 可能复活。

### ADR-DR-16 — claim revision 与 checkpoint 结构关系分离

- `agent_checkpoint.parent_checkpoint_id` 只表示现有的结构/归属关系（尤其 A0-02 subagent boundary → 父 checkpoint），**绝不**承载 import 内容版本链。
- `agent_coverage_claim` 是当前指针与 reservation，`agent_coverage_revision` 是不可变的 per-turn 版本史；incomplete→complete 仅在这两张表前推。`agent_checkpoint.superseded_by` 不纳入 DR-05c-0，避免一个 live whole-transcript checkpoint 被某条 claim 升级时错误隐藏其余 turn 的证据。
- checkpoint list/show 默认继续列出完整 checkpoint 历史；需要“某 turn 当前版本”的 UI/API 必须 join claim/revision 明确呈现。DR-06 的 subagent content 采用同样 source-scoped current/revision 模式，不能复用 parent link 或 whole-checkpoint 隐藏。

### ADR-DR-17 — import 同意、选择器与当前仓库数据边界

- 命令必须显式给出 `--session`、`--path`、`--since` 或 `--all` 之一；无 selector 拒绝，`--all` 不能由默认行为隐式触发。目录发现只可读取安全的文件名/受限 metadata，确认前不得读取 transcript 内容或执行 export。
- TTY 下在首次内容读取前显示 agent、当前仓库范围、候选数/上限、脱敏后写入 traces 及可能 `agent push` 上传的提示，要求确认；非 TTY/`--json` 必须显式 `--yes`，否则稳定错误退出。`--yes` 只跳过此同意，不放宽 provider-root、repo identity、size、time 或 platform 授权。
- 仅导入 working_dir/repository identity 与当前仓库匹配的 session；缺失、歧义或跨仓库的 cwd 一律 skip/quarantine 并给不含路径的可操作诊断。批量操作按 session 原子、批次 best-effort，输出 sanitized 的 `imported|noop|partial|skipped|failed` 汇总及 stable code；不提供跨 session 的假原子性。
- 发现、`--all` 与 `--since` 受 max-entries、时间窗、总字节和 session `--limit` 共同约束；默认 `--limit=20`、单次硬上限 100 session、累计原始输入硬上限 64 MiB（并继续受每源 16 MiB cap 约束）。继续处理必须显式 cursor/下一次调用，不能因为长历史无限扫描。
- 若某 session 在读取/解析/写入时发生非策略性失败，batch 必须以 stable partial-result error 非零退出，同时保留已完成 session 的 sanitized result；仅 tombstoned、非当前仓库或用户未选择的候选可计为显式 `skipped`，不得把真实失败伪装成成功。

### ADR-DR-18 — agent list 的显式 schema 协商

`agent list` v1 是已发布的严格 JSON 合同，而不仅是 `DeclaredAgentCaps` 的 8-bool 合同。实现必须保留默认 v1 的 `ListOutput`/`ListAgentRow` 完整形状；新增 `--schema-version=2` 后才可返回 `capabilities.methods[]`、availability 和 reason。v2 的 method 名、状态枚举、未知 method 忽略规则与 error shape 必须在用户文档、machine output schema 和 compat fixture 中固定；不支持的 schema version 返回稳定 usage error，不能悄悄回退或让字段按 agent 特例漂移。

### ADR-DR-19 — tombstone 是并发写屏障

- erase 事务按稳定 `(agent_kind, provider_session_id)` 写/锁 tombstone，删除 session/claims/identity/export job，并使关联 reservation 的 fence 全部失效；若已有 ref/catalog，保留现有 erase 的可恢复处理。
- 创建/复用 session、预留 claim、推进 import identity、推进 export generation、最终 ref/catalog/claim/revision 事务都必须在**同一事务**确认 tombstone 不存在。任何 0-row/fence 失败都只能 no-op/返回 erased，不能重新创建 session 或推进 ref。
- `--restore-erased` 必须显式确认、写 audit，并在删除 tombstone 后由新的 import attempt 获取新 fence；旧 holder 永远不能借此复活。测试必须调度 erase 与 parse、reservation、对象构造、最终提交四个窗口并证明无删除后可达 checkpoint。

### ADR-DR-20 — 捕获数据可视化（`libra agent graph`）是只读投影

- `libra agent graph <session>` 是**捕获表之上的只读投影**：数据源只有 `agent_session`、`agent_checkpoint`、`agent_import_tombstone`（只读 `erased_session_id` 判定 `erased`）及本计划新增的 `agent_coverage_claim`/`agent_coverage_revision`（M1）与 subagent content/boundary（M5）。**禁止**复用 `libra graph` 的 `ProjectionResolver`/`ThreadBundle`（其以 `thread_id` 加载 `ThreadBundle`，再经 `intent_id → plan/task/run` 关联查询 `ai_index_*`）thread 投影，也不得为 capture 合成 `thread_id`——二者是不同数据模型、不同主键。只复用 TUI **外壳**（`tui_init`/`tui_restore`、ratatui、graph 主题与 pane 布局），不复用其数据层。
- **纯读、零写**：graph 不获取 lease、不改 claim 状态机、不写对象/ref/catalog，不触发 discovery/import/export，且不 INSERT/UPDATE/DELETE 任何 capture 或 tombstone 表。它只按下方**列白名单** SELECT capture/tombstone 表（不 SELECT 未脱敏文本列，因既有列并非全部可靠脱敏），因此不受 ADR-DR-09/10 写路径约束；命中 tombstone 时显示 `erased`，**绝不**重建 session（ADR-DR-19）。
- **渲染模型**：session 根 → 按 `logical_turn_key`/ordinal 排列的 turn 节点；每个 turn 显示 current revision + `agent_coverage_revision` 历史、`completeness` 与 coverage 状态。一条 whole-transcript committed checkpoint 可同时支撑多个 turn，**必须**如实呈现共享关系，不得因某 turn 升级把共享 checkpoint 画成 superseded 或隐藏（ADR-DR-16）。subagent content checkpoint 作为子节点，标注 `link_state`（Codex 可 resolved、Claude 常 unresolved，ADR-DR-05）。区分 `scope`(temporary|committed|subagent) 与 `source_channel` provenance。
- **Legacy capture contract**：没有 `agent_coverage_claim`/`agent_coverage_revision` 的既有 session（M1 之前捕获）必须以 `coverage_state="unindexed"` 呈现为按 `created_at` 排序的 checkpoint chronology；不得伪造 turn/revision、读取 transcript blob 或写回填。JSON 同样显式标注该状态。
- **隐私（GC-DR-13）——显式列白名单**：capture 元数据里含未脱敏字段（`agent_session.metadata_json` 存原始 `transcript_path`，`working_dir` 直接来自 hook 信封，`agent_checkpoint.description` 是自由文本，checkpoint metadata 也含 `working_dir`，且 runtime 明示其 redactor 不保证清除任意绝对路径）。“只渲染已脱敏 metadata”不是充分边界。因此 graph 只能读**固定列白名单**（其余列一律不进 SELECT）：
  - `agent_session`：仅 `session_id`、`agent_kind`、`state`、时间字段（`created_at`/`updated_at` 等）。
  - `agent_checkpoint`：仅 `checkpoint_id`、`session_id`、`parent_checkpoint_id`、`scope`、`created_at`（结构/归属列）。
  - `agent_coverage_claim`/`agent_coverage_revision`：仅 `logical_turn_key`、`coverage_schema_version`、`completeness`、coverage/claim `state`、`revision`、`checkpoint_id`、`source_channel`、时间字段（`ordinal` 非持久化列，由展示位置派生，见 JSON v1 契约）。
  - subagent 关联：仅 `link_state` 与结构外键。
  - **禁止**读取或输出 `agent_session.working_dir`、`agent_session.metadata_json`、`agent_checkpoint.description`、`redaction_report`、`coverage_digest` 原文，**禁止**打开 `metadata_blob_oid`/`tree_oid`/`traces_commit`/`parent_commit` 指向的对象，**不**渲染任意 description/metadata/自由文本。JSON 复用同一白名单。
- **输出模式**：`--json`/`--machine` 是**全局根 flag**（由 `OutputConfig::resolve` 解析为 `json_format`，非 `GraphArgs` 字段）。有 `json_format` 时走 `emit_json_data`；无 `json_format` 时若非 TTY，**必须在初始化 TUI 前以稳定 usage error 拒绝**并提示使用 `--json`（`emit_json_data` 在无 active JSON mode 时会内部 error，故不能对普通非 TTY 调用凭空套用）。不得隐式改变全局输出格式。TUI 初始化本身也拒绝非终端 stdin/stdout。`--machine` 与 `--json` 走同一分支，测试须各覆盖一次。
- **JSON v1 契约（完整机器 schema）**：`emit_json_data("agent_graph", …)` 的 `data` 固定为：
  - 顶层 `{ schema_version: 1 (int), state: "present"|"erased", session: <SessionObj>|null, turns: <TurnObj[]>, subagents: <SubagentsObj> }`。
  - `state="erased"` 时：`session=null`、`turns=[]`、`subagents={available:false, unavailable_reason:"erased", nodes:[]}`，且**不含**任何 provider/source/path/metadata 键。
  - `SessionObj = { session_id:str, agent_kind:str, state:str, created_at:int, updated_at:int }`（严格等于 session 白名单列，无 working_dir/metadata）。
  - `TurnObj = { logical_turn_key:str, ordinal:int, coverage_schema_version:int|null, coverage_state:"indexed"|"unindexed", completeness:"incomplete"|"complete"|null, current_revision:int|null, checkpoint_id:str|null, source_channel:str|null, revisions:<RevisionObj[]> }`。
    - `ordinal` 是**派生的展示序号**（返回数组内 0 起的位置，indexed turn 按 `logical_turn_key` 规范序、legacy 按 `created_at`），**不是** claim 表持久化列——coverage 表只持久化 `logical_turn_key`，不持久化 ordinal。
    - **indexed**（有 claim 且**至少一条已提交 `agent_coverage_revision`**）：`coverage_schema_version`、`source_channel`、`completeness`、`current_revision` 均来自 `NOT NULL` 的 claim/revision 白名单列，非空。仅处于 `reserved_*`、尚无已提交 revision 的在途 claim（`checkpoint_id` 可空）**不渲染为 indexed turn**（属正在写入的瞬态，读侧跳过），因此非空承诺恒成立。
    - **unindexed**（M1 前 capture，仅可读 checkpoint 结构列）：`logical_turn_key` 用 `"checkpoint:<checkpoint_id>"` 占位，`coverage_schema_version=null`、`source_channel=null`、`completeness=null`、`current_revision=null`、`revisions=[]`——不得凭空造这些字段。
  - `RevisionObj = { revision:int, completeness:str, checkpoint_id:str, source_channel:str, created_at:int }`。
  - `SubagentsObj = { available:bool, unavailable_reason:str|null, nodes:<SubagentNodeObj[]> }`（M5 前 `available:false`、`nodes:[]`）；`SubagentNodeObj = { checkpoint_id:str, link_state:"resolved"|"unresolved", boundary_checkpoint_id:str|null, created_at:int }`。
  - **禁止**任何自由文本 description、绝对路径、`metadata_json`、blob 内容进入上述任一字段。
  - v1 是冻结形状，compat fixture pin；未来字段演进只经**显式 `--schema-version=2`**（同 ADR-DR-18 的协商模式）返回，v1 default 不变。本切片只交付 v1，不实现 v2。

## 参照基线：`entireio/cli` 的三种数据读取方案

Entire 对每类 agent 抽象出统一接口（`GetSessionDir` / `ResolveSessionFile` / `ReadTranscript` / `ReadSession` / `PrepareTranscript` / `GetTranscriptPosition`），但三类 agent 的底层落盘形态不同，读取策略也不同：

| Agent | 落盘位置 | 读取方式 | 关键源码（entireio/cli） |
|---|---|---|---|
| **Claude Code** | `~/.claude/projects/<project-slug>/*.jsonl`（每 session 一个 JSONL；子 agent 在 `<sessionID>/subagents/`） | hook 递来 `transcript_path` → 整段读 JSONL；**Stop 前轮询等待异步 flush**（`waitForTranscriptFlush` / `checkStopSentinel`）；另有**批量导入器**扫 `~/.claude/projects/**/*.jsonl` | `cmd/entire/cli/agent/claudecode/{hooks.go,lifecycle.go,claude.go}`、`cmd/entire/cli/agentimport/claude.go` |
| **Codex** | `$CODEX_HOME/sessions/YYYY/MM/DD/rollout-...-<session-id>.jsonl`（日期分区） | hook 递来 `transcript_path` → 读 rollout JSONL；**仅有 session ID 时** `findRolloutBySessionID` 回溯；按行类型解析 | `cmd/entire/cli/agent/codex/{codex.go,transcript.go,lifecycle.go}` |
| **OpenCode** | **无稳定可直接读的 transcript 文件**（`.opencode` 是 **project 本地**插件目录；hook 信封无 `transcript_path`） | **`opencode export <sessionID>`** → 缓存/内存 JSON → 读 `ExportSession{Info, Messages}` | `cmd/entire/cli/agent/opencode/{opencode.go,lifecycle.go}` |

**结论：**

- **Codex 与 Claude 高度类似**——hook 指针 + 磁盘 JSONL；差别在目录布局与「按 session ID 发现」。
- **OpenCode 明显不同**——命令桥接导出，不是纯读文件。
- 共同点：**hook 是触发器，数据来自 agent 持久化会话；且都有脱离 hook 的发现/导入路径**。

探针版本声明（互操作）：OpenCode 插件注释已钉 **opencode 1.17.13**（2026-07-05）；Claude/Codex slug/rollout 规则以开工时 Entire 源码 + 本地探针为准，变更时加测试向量而非静默猜。

## 横切工程约束（GC-DR，本计划全部任务生效）

- **GC-DR-01 现状核实**：动手前 `rg` 复核锚点；已实现则降级为补测/补文档。
- **GC-DR-02 授权不降级**：任何新读路径（发现、import `--path`、export Bytes）不得弱于现有 provider-root 门；fail-closed。
- **GC-DR-03 redaction-before-persist**：见 ADR-DR-03/12；测试须覆盖「export 伪密钥不落 traces、SQLite、日志和错误文本」。
- **GC-DR-04 字节与时间预算**：
  - Path 读：统一消费 `min(agent.max_transcript_read_bytes, adapter_hard_cap)`；默认有效 cap 保持 16 MiB，避免本计划静默把 hook 内存上限放大 16 倍。若未来提升默认值，必须有内存/延迟基准和发布说明。compliance 配置高于 hard cap 时诊断实际有效值，禁止“配置看似生效但被静默忽略”。
  - Bytes/export：独立 cap，默认 16 MiB；stdout 采用流式有界读取，超限立即终止子进程并清理。
  - OpenCode 单次 export 子进程 wall-clock 默认 ≤3s（可配置）；整链路 end-to-end 须落在 hook 10s 内，否则跳过内容、仅 metadata + warn。
  - **OpenCode hook timeout 前提**：当前插件模板未显式写 timeout，10s 为假设上限。DR-04b 任务卡必须核实/配置插件 timeout；若上游 hook timeout < 10s，则 export 预算须相应收紧并文档化。
  - DR-01 flush-wait ≤2s 且为后续 write 留余量。
- **GC-DR-05 迁移注册**：凡新表/新列必须 `sql/migrations/*` + `_down.sql` + `builtin_migrations()` 注册 + `tests/db_migration_test.rs` 钉死列表；**目录落盘 ≠ 生效**。
- **GC-DR-05a 迁移兼容与回滚**：writer 启动前校验 schema；older schema 自动升级失败则 fail-closed，newer schema 复用既有 schema-newer 错误策略。down migration 只承诺开发/测试回滚；一旦新 checkpoint 使用新 metadata/claim 契约，生产回滚旧二进制前必须先停止 writer、备份 DB/refs，并验证旧二进制不会误读。禁止“删表即回滚”的数据丢失方案。
- **GC-DR-06 CLI 表面门禁**（DR-05/DR-07）：`cli.rs`/`command/agent/mod.rs` 注册、`AGENT_EXAMPLES`/`after_help`、用户文档 `docs/commands/agent.md`（EN）+ `docs/commands/zh-CN/agent.md`（ZH）+ **活跃开发契约 `docs/development/tracing/agent.md`**（`docs/development/commands/agent.md` 仅作索引链接、不重复规范；compat guard `matrix_alignment` 校验 tracing 路径）+ `COMPATIBILITY.md`（`agent import` 与 `agent graph` 均为 `agent` 子命令、Libra-only intentionally-different，作为新章节并入现有页面，而非新建顶层命令页）、`tests/INDEX.md`、`docs/error-codes.md`、compat help/examples 守卫。**新错误码**：`docs/error-codes.md` 现有 `LBR-AGENT-001..010`，本计划新码从 **`LBR-AGENT-011`** 起连续分配（至少覆盖：跨仓库归属冲突、缺失/歧义 working_dir、不支持的 schema version、partial-batch 失败、命中 tombstone/erased、Bytes 授权标签缺失）；每新增一个 `StableErrorCode` 变体必须同步 `docs/error-codes.md`（`compat_error_codes_doc_sync` 守卫）。`agent list` v1 形状不得变化（当前 `AGENT_LIST_SCHEMA_VERSION=1`、`ListArgs {}` 无参数）；新增 `--schema-version` flag 与 v2 输出为**纯增量**，v2 schema flag、unknown-version error、methods/availability 的 machine schema 和 fixtures 必须一并登记。

  - `agent_import_tombstone` 必须在 `(agent_kind, provider_session_id)` 建 UNIQUE（导入阻断键），并在 `erased_session_id`（`TEXT NOT NULL UNIQUE`，无外键）建 UNIQUE 供 DR-07 只读 `erased` 查询；`source_fingerprint` 只作审计列。

- **GC-DR-12 状态机与清理**：
  - claim/revision 行不得无限累积：session erase、`clean --gc`、retention pass 必须物理删除 claim、revision 与 identity 行；但 erase 前先写 tombstone，并以 ADR-DR-19 作为全部 writer 的事务内写屏障。

- **GC-DR-13 日志与诊断隐私**：错误和 tracing 只记录 agent kind、稳定错误码、大小、耗时、摘要前缀；默认不记录 transcript、prompt、tool input、绝对 source path、session export stdout/stderr。子进程 stderr 也必须 cap + redact 后才能显示。
- **GC-DR-14 平台与标准**：路径安全实现须覆盖 Linux/macOS/Windows；无法提供等价 no-follow/beneath 语义的平台应明确 fail-closed 或将 import 标为 unsupported，不得悄悄退化为字符串 `starts_with`。

### DR-04b — OpenCode export 桥接（依赖 DR-04a + DR-05c-0）

- **触发**：`session.idle` → `TurnEnd`（**不能**改成仅 SessionEnd；headless 下 deleted 不可靠）。
- **行为**：每 idle **整份** `opencode export <sessionID>` → seam `Bytes` → typed normalize/redact → splitter → 对每个 turn 执行 claim 决策；未覆盖则 append，等价/更完整已覆盖则 no-op，incomplete→complete 则原子前推该 turn 的 revision。
- **不引入** job 队列 / 常驻 worker。
- **`agent_export_job`**（专用小表）：按 ADR-DR-11 使用 observed/processed generation、owner/fence、有界循环和 TTL；崩溃后由后续 idle/import/启动 scavenger backstop。
- **子进程安全**：复用 `trust.rs`（绝对路径 + sha256 + inode/mtime，漂移 `LBR-AGENT-005`）；**不**走 `sh -c` sandbox helper；`SandboxEnforcement::Required`；网络禁用；argv 结构化；session ID 校验；输出 cap + deadline（GC-DR-04）。
  - **信任边界说明**：`trust.rs` 保证的是「运行的 `opencode` 二进制未被篡改」；sandbox 保证的是「子进程权限受限」。二者都不能让恶意 exporter 输出变可信。因此 `ExportAuthorized` 标签只表示「字节来自本进程刚调用的受信任 exporter」，writer 仍须对解析结果做 schema 校验、size cap、redaction，禁止把 exporter 输出当天然安全。
  - **最小 sandbox profile**：只读 binary/动态库/OpenCode session store/必要配置；仅专用 temp 可写；env allowlist；cwd 不进入项目插件目录；显式禁用插件/hooks；禁止网络。任务卡必须用受支持 OpenCode 版本证明离线 export 在该 profile 下可运行；若不能，先停在 capability unavailable，不得降级执行完整未隔离 CLI。
- **性能诚实记账**：
  - 每 idle：+1 次 export 子进程 + 全量 parse/redact/hash（与既有 TurnEnd 全量处理同阶）
  - coverage 门：减少的是**重复 turn 的 VCS append 次数**，不是单次 blob 体积
  - 长 session 存储若每 turn 仍存整份 transcript，累积仍 O(n²) 字节——**本计划不改为增量 blob**（避免扩大 scope）；记入风险登记
- **Scope**：L  
- **Verification**：  
  `opencode_export_argv_no_shell`、`opencode_export_rejects_bad_session_id`、`opencode_export_binary_trust_revalidates`、`opencode_export_inflight_generation_merges_third_idle`、`opencode_export_max_loop_preserves_dirty`、`opencode_export_stale_owner_cannot_release_or_commit`、`opencode_export_inflight_ttl_takeover`、`opencode_export_whole_session_idempotent`、`opencode_export_bytes_path_byte_cap`、`opencode_export_oversize_session_degrades`、`opencode_export_offline_sandbox_profile`、`opencode_export_plaintext_never_in_persist_or_logs`（密钥夹具）。

### DR-05c — import identity + 注入式 writer + fence

- **表** `agent_import_identity`：稳定键 `(agent_kind, provider_session_id, source_kind, source_id, schema_version)`（不含 content digest，source_id 不泄露绝对 home）；字段至少包含 observed/committed digest、attempt id/checkpoint、next ordinal、state(discovered|leased|writing|partial|committed|failed)、owner/lease/fence、last_error_code、updated_at。

- **CAS 循环内 same-ID 叶 no-op**；transaction-injected ref CAS 同时验证 identity + claims fence，最终同事务提交。

- import 只补未覆盖洞或完成合法的 incomplete→complete 升级；等价 complete 内容复用先提交 checkpoint，内容冲突进入 doctor，不再使用“live 无条件权威”口号。

## 执行检查必备需求（强制）

以下要求适用于本文每一个改进项。任一要求未满足，该改进不得标记完成，不得进入下一项发布流程。本节按用户要求在本文中明文保留；语义与 `plan-20260708.md` 的 ER-01..ER-11 对齐，执行时不另建第二套发布、review 或重试流程。

1. **每一个改进验收必须三项全部通过**：`cargo +nightly fmt --all --check` 无格式差异；`cargo clippy --all-targets --all-features -- -D warnings` 无警告；`source .env.test && cargo test <指定测试用例>` 全部通过。任务卡里的指定测试用例必须在开工前写清楚，不能用泛泛的 `cargo test` 替代。
2. **每一个改进都需要交由 Codex 进行 review**：实现和本地验收完成后，必须让 Codex 做代码 review；根据 review 结果继续修改并重新执行验收，直到 Codex review 明确通过。
3. **每一个改进都必须先做文档与代码核对**：执行前核对相关文档和当前代码实现，根据核对结果把要完成的任务调整到最新状态，然后再开始实现；如果代码已经完成、文档滞后或计划过期，先把任务改为补测试、补文档、降级非目标或继续实现的最新状态。
4. **本地版本管理使用 Libra**：本仓库本地使用 Libra 作为版本管理，不要当做 Git 仓库处理；状态、暂存、提交、推送都使用 `libra` 命令和 Libra-native 工作流。
5. **所有改进都在 `main` 分支进行**：不要创建 worktree，不要创建 feature branch；开工前确认当前分支是 `main`，并在脏工作区中只处理当前改进相关路径。
6. **每一个改进完成后发布 patch 版本并推送**：每一个改进验收和 Codex review 通过后，对当前版本 patch 加 1（以开工时 `Cargo.toml` 实际 `version` 为准，如 `0.18.NNN` → `0.18.(NNN+1)`，不要硬编码某个历史号）；`worker` 和 `web` 子目录下的 `package.json` 版本必须同步保持一致。不手工修改 `Cargo.lock`；版本修改后执行 `cargo build --release`，让工具链进行需要的锁文件更新。构建出的 release 版本 `target/release/libra` 必须拷贝到 `$HOME/.libra/bin/libra`。随后调用 Libra 命令执行 `libra add <相关路径>`、`libra commit -s -m "<scope>: <summary>"`、`libra push origin main` 推送到 GitHub。**不要使用 `commit -a`**：本仓库工作区常有并发节点改动与带外删除，`-a` 会把非本任务路径一起提交（与第 5 条「只处理当前改进相关路径」冲突）；一律用显式 `libra add <相关路径>` 后 `libra commit -s -m`（不加 `-a`）。
7. **每一个改进都必须同步用户命令文档**：配合更新 `docs/commands` 下对应英文文档和中文文档；如果对应命令没有文档，则创建中英文两份文档，并同步开发文档、兼容矩阵和测试索引。

## 本计划专属执行控制（补充，不得削弱强制检查）

上节是唯一强制发布、review、push 与服务错误流程，本节只补本计划的执行边界，**不得覆盖或放宽上节任一要求**。

  - **DR-05** import：对三个 agent 的真实历史会话执行 `libra agent import`（显式 selector + 同意），验证当前仓库归属、per-turn 幂等、跨仓库拒绝与 erased 屏障。
  - **DR-06** 子 agent：用真实 Codex（原生发 subagent 边界）与真实 Claude（不发边界、磁盘 `subagents/`）验证 boundary/content 分型与 Claude 常态 `unresolved`。
  - **DR-07** graph：对上述真实捕获数据运行 `libra agent graph <session>`，人工核对 TUI 渲染与 `--json` 输出，并实测白名单不泄原文/绝对路径、`erased` 显示、非 TTY 拒绝与只读零写。

  - **发布窗口计入（与 3-smoke 规则对齐）**：pre-review 的 live gate 证据**不直接**计入发布窗口三份记录。只有在**对应 commit 成功公开 release 之后重新运行** live gate，并记录 `release/tag`、`commit`、UTC 时间、独立记录 ID 与 sanitized result 时，该次 post-release 运行方可计为「3 份独立 smoke/错误率记录」之一；每份 live record 最多计一次，pre-review 证据不得挪用。
  - **证据 schema 与存放（sanitized-only）**：每次 live 运行只留存固定字段——`release/tag`（pre-review 则标 `pre-review`）、`commit`、UTC 时间、provider 名与版本、稳定结果码、聚合计数（session/turn/checkpoint/subagent 数等）——写入指定证据文件（`docs/development/plan/evidence/agent-live-gate-<tag-or-commit>.md`）。持久化前必须 redact；**禁止**留存原始 stdout/stderr、transcript、prompt、tool input、session 路径或任何绝对路径（延续 GC-DR-13；CLI stderr 也须 cap+redact 后方可记录）。
- **文档与接口门**：DR-05 同步 `docs/commands` 中英、开发文档、`COMPATIBILITY.md`（Libra-only 说明）、`tests/INDEX.md`、`docs/error-codes.md` 和 help/examples guard；`agent list` 同时维护 v1 fixture 与 v2 协商契约。确定性用例优先 `test-provider`，子进程/文件使用注入 fake，不把商业 API key 当默认验收前置。
- **发布/重试解释**：严格采用上节第 6、9、10 条；特别是本次 `push` 失败只记录、不重试，其他暂时性内部服务错误按 10 分钟等待后重试同一步。认证、权限、参数或确定性代码/配置错误转为当前切片修复并重新验收，不能用“有界重试”替代。**例外**：当 `push` 失败是 **origin 版本领先导致的 non-fast-forward**（并发抢号）时，按下一条「版本抢号与 push 冲突解决」处理——那是「整合后推送新状态」，不是重试同一次 push，与第 9 条不冲突。
- **版本抢号与 push 冲突解决（与 `plan-20260708.md` 并发共享 `main`/版本号）**：`plan-20260708.md` **正在另一台机器并发执行**，与本计划**争夺同一个 `Cargo.toml` 单调版本号**并推送同一 `origin/main`。本计划据此采用**抢号模型**：版本号在 **push 成功时**才算领到，不在本地 commit 时锁定;遇到 origin 已领先就整合、重新领号、再推。**`plan-20260708.md` 已镜像本节协议与「执行检查必备需求（强制）」**，两台机器遵循同一套发布/重试/抢号语义（双向权威、内容一致）。
  - **push 冲突流程**：当 `libra push origin main` 因 origin 版本更高/领先而失败（non-fast-forward，如 `LBR-CONFLICT-002`）时：
    1. `libra pull`（或 `fetch` + `merge`）整合 origin 的并发提交（本仓库 pull 有历史 hang 记录，须加超时/后台监控，超时则报告不静默卡死）;
    2. **按语义解决冲突**（见下表），不做纯文本合并;
    3. 合并后**重新领号并重跑验收**：版本改为 `max(origin, 本地) + 1`、`worker`/`web` 的 `package.json` 三处同步一致、`cargo build --release` 让工具链更新 `Cargo.lock`，再跑 `cargo +nightly fmt --all --check`、`cargo clippy --all-targets --all-features -- -D warnings` 与本切片指定测试，确认合并未破坏构建/验收;
    4. 再 `libra push origin main`;仍失败则重复本循环（用 tight `libra pull origin main && libra push origin main` 收窄竞态窗口）。**禁止 `--force`**——绝不丢弃 origin 的并发提交。
  - **语义冲突解决策略**（共享登记/文档文件）：
    | 文件 | 语义解决 |
    |---|---|
    | `Cargo.toml` `version` + `worker/package.json` + `web/package.json` | 不做文本合并;取 `max(两侧) + 1` 为新号，三处同步一致，`cargo build --release` 更新 `Cargo.lock` |
    | `docs/error-codes.md` / `LBR-*` 码 | 取并集;若两侧用了**同一码号**，把本地的重排到下一个空闲码并更新全部引用（`src` + `docs` + `tests`） |
    | `sql/migrations/*` + `builtin_migrations()` + `tests/db_migration_test.rs` 钉死列表 | 取并集并保序;文件名日期/迁移 ID 冲突则给本地迁移重命名，再同步注册与钉死列表 |
    | `tests/INDEX.md` | 并集两侧 Wave 条目 |
    | `COMPATIBILITY.md` | 并集行;同一命令行冲突则合并 Notes |
    | `src/cli.rs` 命令组 / `command/agent/mod.rs` | 并集命令变体 |
    | 其它源码 | 通常不同域（Git 兼容 vs agent 捕获），按各自意图合并;真正同区冲突按行为正确性解决，不机械取一侧 |
  - **与第 6、9 条的关系**：本条**细化**而非放宽——第 9 条"push 失败不 blind-retry"仍适用于真正的 push 失败（认证/权限/网络/服务端：记录、不重试、随下次发布携带）;本条只处理 **origin 版本领先的 non-fast-forward**，"pull + 语义合并 + 重新领号 + 推送新状态"正是第 9 条要求的"修复→重新验收→版本构建后再尝试"，非重试同一 push。第 6 条"读开工实际版本 +patch"照旧，只是抢号下"实际版本"在每次冲突后**以 pull 后的 origin 值重新读取**，据此领下一个号。

### 里程碑验收与回滚

每个里程碑除任务指定测试外，必须运行 `cargo test --all`；涉及 OpenCode/test-provider 的目标按 `tests/INDEX.md` 使用正确 feature/env。M1/M3/M4/M5 还必须运行 migration up/down、new-binary→old-binary 行为测试和故障注入矩阵。每个里程碑还须完成「本机 live agent 执行验证门」——按该门的逐 DR「已交付、可读取」provider 映射，用相应本机真实 CLI 端到端跑通该里程碑所含 DR 并留存 sanitized 证据——方可宣告 M 级完成；不要求尚未交付的 source path（如 DR-04b 之前的 OpenCode 内容源），三 CLI 完整矩阵仅在三条读取路径均交付后整体强制。

- **上线前**：备份 `.libra/libra.db`、记录 `refs/libra/traces` head 与 schema version；验证 down migration 不删除用户唯一数据。
- **上线中**：先升级 schema，再启用新 writer；不允许新 writer 在旧 schema 上“尽量运行”。
- **回滚**：先停止 hook/import/export writer，再恢复兼容 binary/schema；若已有新格式 checkpoint/claim，优先前滚修复，不盲目 down migration。回滚后必须验证 traces 可达性、catalog/claim 一致性和 tombstone 仍生效。

### 故障恢复矩阵

| 故障点 | 可接受残留 | 恢复动作 | 禁止结果 |
|---|---|---|---|
| reservation 后、对象前 | 过期 reservation | lease takeover → abandoned/重试 | 永久阻塞 turn |
| 对象后、最终事务前 | 不可达对象 | 后续 object GC | 推进 ref 或重复可见叶 |
| 最终事务提交成功、响应前 | 完整 catalog/claim/ref | 重试按逻辑键 no-op | 第二次 append |
| live 抢占 import 后旧 holder 回写 | 不可达对象 | fence 拒绝所有 writeback | 陈旧 holder 更新 ref/identity |
| traces ref CAS 发现 head 已变 | 本轮不可达对象 | 回滚最终事务、重读 head、重建 commit 后在 lease 内重试 | 将旧 parent 的 commit 接到新 head 或跳过 claim/tombstone 复核 |
| export 达 max loops | dirty job | 下次 idle/takeover 继续 | 错标 clean 丢 generation |
| DB/schema 不可用 | 可重放请求/错误 | 修复 DB 后重放 | 无 claim 直接 append |
| erase 与 parse/reservation/对象构造/最终提交并发 | tombstone + 不可达对象 | tombstone/fence 条件失败后 no-op 或 erased；旧 holder 不得重建 session | 删除后产生可达 checkpoint、claim、identity 或 export job |

## 性能与容量摘要（避免误读）

| 操作 | 单次成本 | 随 turn 数 n 累积 |
|---|---|---|
| 读 + redact + hash（一切 agent，既有） | O(当前 transcript 字节) | 每 TurnEnd 一次 → CPU **O(n²)** |
| VCS append（既有，每新 turn 一次） | O(当前 transcript 字节) 写 blob | 存储 **O(n²)** 若每叶存全量 |
| coverage 门命中 | O(1) DB | 避免**重复** append，不缩小单叶体积 |
| OpenCode export（本计划新增） | 子进程 + 全量 JSON | 每 idle 一次；受 cap/deadline 约束 |

本计划**优化的是正确性与入口覆盖 + 重复写抑制**，不是 transcript 存储模型重做。
